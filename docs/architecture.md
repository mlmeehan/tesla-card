## 1. Executive Summary

`tesla-card` is a Tesla-app-style custom [Lovelace](https://www.home-assistant.io/dashboards/)
card for Home Assistant, built in **TypeScript (strict) + [Lit](https://lit.dev) 3**. It renders a
single vehicle as a centred **living hero** (recolorable inline-SVG car with charge/aperture state)
over a **tab bar** of feature panels (charging, climate, energy, closures, tires, location, media),
plus quick-action toggles and command buttons. Beyond the vehicle card it also ships a set of
standalone **ecosystem cards** (solar / powerwall / grid / home / wall-connector / **generator**) and a
composed **"My Home" energy Scene**, all from the same bundle.

Three capability arcs define the current shape:

- **Live, honest energy flow.** One sign/unit convention and a single balance authority drive the
  Scene's Gateway bus, while the Hero's charge halo reads the discrete `charging_status` entity — a test
  asserts the two agree, so the visuals never disagree with the numbers or animate a false flow (Epics 4/6/8).
- **No-YAML Enhanced Configuration (Epic 9 / the 1.0.0 net-new).** An HA-native GUI editor
  delivers a guided first-run wizard + a normal form for theming, per-entity overrides, and "My Home"
  Scene customization (hide / reorder / cross-row promote / duplicate nodes), plus a brand-new
  `generator` node **type** — all additive and zero-diff-when-absent (§12).
- **Post-1.0 hardening (Epics 10–17, all off the 1.0.0 critical path; `v1.0.0` shipped 2026-07-12).**
  Render polish (Epic 10's Scene-aware editor for `custom:tc-my-home`, §12.7; Epic 11's composed-Scene +
  embedded-cell polish, §8.1), **orthogonally-routed Scene bus legs** (Epic 13, §8), a full
  **per-integration dialect** arc — name-alias entity resolution, vehicle-device-scoped detection, and
  end-to-end charging classification across Tesla Fleet / Teslemetry / Tessie / `tesla_custom` (Epics
  14–16, §5.5) — and Epic 17's deferred-ledger flush (charging-surface + non-uniform-geometry hardening).

It ships as a **single bundled ES module** (`dist/tesla-card.js`) consumed by Home Assistant as a
Lovelace resource and distributed via [HACS](https://hacs.xyz/). There is **no application server,
no API layer, and no datastore** — all state comes from the HA `hass` object injected by the
dashboard at runtime, every service call goes back through `hass.callService`, and the one historical
read rides `hass.callWS` (HA's own authenticated WebSocket — never a socket of the card's own).

The card's robustness rests on **entity resolution by stable function-name** — not hard-coded IDs — so it
works across installs, device renames, and integration prefixes (§5).

## 3. Layered Architecture — `data/ ← flow/ ← components/`

The source is organized into three layers with a **one-way dependency direction**, enforced by the
`no-cycle` and `no-bare-hass-states` lint gates:

```mermaid
%%{init: {"flowchart": {"wrappingWidth": 500}}}%%
flowchart TB
    hass[("hass.states")]
    data["<b>data/</b> — freshness · dialect · energy · history · platforms · registry · resolve · slice · wake<br/>pure read-model + normalization boundary (9 modules)"]
    flow["<b>flow/</b> — balance · model · binding · renderer · scene-bus · my-home · instances<br/>pure energy-flow math (7 modules); imports data/, never components/"]
    components["<b>components/</b> — tc-* + helpers<br/>hero / panels / quick-actions / commands / ecosystem / scene + car / weather / chart / node-hero<br/>Lit elements + render helpers; import flow/ and data/"]
    hass -->|"read by data/ — the ONLY layer allowed to"| data
    data -->|"canonical, sign-normalized data"| flow
    flow -->|"FlowModel + visual derivations"| components
    classDef dataL stroke:#3987e5,stroke-width:3px
    classDef flowL stroke:#199e70,stroke-width:3px
    classDef compL stroke:#c98500,stroke-width:3px
    class data dataL
    class flow flowL
    class components compL
```

- **`src/data/`** is the data-access boundary (9 modules). It is the **only** subtree allowed to read
  `hass.states`/`.entities`/`.devices`. `data/freshness.ts` owns the universal read-model
  (`read`/`readKey`/`readRaw`, `referenceNow`, staleness classification, `isQuiescent`); `data/resolve.ts`
  + `data/energy.ts` resolve entities; `data/dialect.ts` quarantines per-integration spelling
  (`normalizeChargingState`/`normalizeCoverState`/`normalizeLockState`, the `DIALECTS` table, dialect
  detection + name-aliasing, the raw→canonical power-sign flip); `data/platforms.ts` is a leaf holding the
  shared `TESLA_PLATFORMS` set (so `resolve.ts` and `dialect.ts` co-import it without a cycle — Epic 14);
  `data/registry.ts` holds the role vocabulary; `data/wake.ts` the wake gate;
  `data/slice.ts` the reflow gate; `data/history.ts` reaches HA recorder history over
  `hass.callWS('history/history_during_period')` (**not** `hass.states`, **never** polled) so the boundary
  still holds. Every `data/` module imports only root leaves + sibling `data/` — **zero upward imports**.
- **`src/flow/`** is pure energy-flow logic (7 modules). It imports `data/` (and root leaves) and **never**
  `components/`. `flow/balance.ts` owns the single sign/unit convention; `flow/instances.ts` (Epic 9)
  is the DOM-free per-instance identity helper.
- **`src/components/`** holds the Lit elements (`tc-*`) and pure render helpers (`car.ts`,
  `weather-vignette.ts`, `chart.ts`, `node-hero.ts`). It imports `flow/` and `data/`.
- **Root leaf modules** (`strings.ts`, `log.ts`, `const.ts`) import nothing upward, keeping the cycle
  check trivially green. All user-facing copy lives in `strings.ts`; all `console.*` routes through
  `log.ts` (a single neutral `[tesla-card]` prefix, no brand colour).

> **The editor is the one sanctioned boundary exception (D7):** `src/editor.ts` MAY read its **own**
> `hass` for discovery — the card **runtime** never does. Guardrails and the self-invalidating
> `no-bare-hass-states` baseline are in §12.2.

## 4. Component Tree (the `components/` layer)

A single top-level custom element orchestrates a flat set of children. There is **one**
`@customElement('tesla-card')`; every child is a `tc-*` element and (with a few exceptions) extends a
shared base class. Components communicate **downward** by props and **upward** by `CustomEvent`.

```mermaid
flowchart TD

    hero["<b>tc-hero</b><br/>components/hero.ts<br/>living car (paint + charge + apertures)<br/>+ battery + status/lock line"]
    qa["<b>tc-quick-actions</b><br/>components/quick-actions.ts<br/>optimistic toggles<br/>(lock/climate/port/sentry/…)"]
    cmds["<b>tc-commands</b><br/>components/commands.ts<br/>fire-and-forget command buttons<br/>(wake/honk/flash/…)"]
    panels["<b>tc-panel-*</b><br/>the active tab —<br/>exactly one in the DOM at a time"]

    card --> qa
```

*Dashed nodes are the tab panels — exactly one is in the DOM at a time.*

**Standalone ecosystem cards** (own dashboard cards; share the `EcosystemCard` shell; use
`chart`/`node-hero`):

```mermaid
flowchart LR
    shell["<b>EcosystemCard</b> shared shell<br/>components/ecosystem-card.ts<br/>(uses chart / node-hero helpers)"]
    shell --> solar["tc-solar"] & pw["tc-powerwall"] & grid["tc-grid"] & home["tc-home"] & wc["tc-wall-connector"] & gen["tc-generator"]
```

Two more standalone elements round out the bundle:

- **`tc-my-home`** (`components/my-home.ts`) — the "My Home" Scene element: live SceneBus render
  of the energy Scene (+ embedded vehicle cells).
- **`tesla-card-editor`** (`src/editor.ts`) — LitElement, implements `LovelaceCardEditor`
  (lazy-loaded).

**Element count.** The bundle defines **20** custom elements total: `tesla-card`,
`tesla-card-editor`, and **18 `tc-*`** elements. Of those, **8 are Lovelace picker cards** registered
in `window.customCards` — `tesla-card` + `tc-solar` / `tc-powerwall` / `tc-grid` / `tc-home` /
`tc-wall-connector` / `tc-generator` / `tc-my-home`. **Editor registration timing differs by
artifact:** at the **source entry** (Vitest, Vite dev) the editor is **not** registered at load — the
dynamic import is genuinely lazy, pinned by `tesla-card.contract.test.ts`
(`editorDefinedAtBundleLoad` must be `false`) — but in the **shipped bundle**, `inlineDynamicImports`
hoists the editor module, so its registration executes at bundle evaluation (all 20 defines observed
when loading `dist/` in this scan). What stays lazy everywhere is editor *instantiation*: no
`tesla-card-editor` node exists until `getConfigElement()`. `car.ts`, `weather-vignette.ts`,
`chart.ts`, `node-hero.ts`, and `ecosystem-card.ts` register **no** element (render helpers / abstract
base class).

**Base class — `TcBase` (`src/base.ts`).** Extends `LitElement` and supplies exactly two reactive
properties to every vehicle child:

```ts
@property({ attribute: false }) public hass?: HomeAssistant;
@property({ attribute: false }) public config!: TeslaCardConfig;   // the RESOLVED config
```

Children read `config` (not `_config`). The elements that don't extend `TcBase` directly are
**`tc-slider`** and **`tesla-card-editor`** (extend `LitElement` directly), and the **ecosystem cards** +
**`tc-my-home`**, which extend `EcosystemCard` / `LitElement` because they are standalone surfaces, not
vehicle children.

**Registration.** Children are pulled in by **side-effect imports** in `src/tesla-card.ts`
(`import './components/hero';` …) — each imported module runs its own `@customElement(...)` on load.
`tc-slider` is side-effect-imported by its consumers (`panel-charging.ts`, `panel-media.ts`,
`powerwall.ts`), not by the parent. **`tc-my-home` independently side-effect-imports all six ecosystem
children** (incl. `tc-generator`) because the Scene can mount standalone — leaning on the `tesla-card`
entry's imports would leave a present node rendering a blank element. Assert registration against the
**bundle entry** (`tesla-card.contract.test.ts`), not a convenient internal module.

**One panel in the DOM.** The parent renders **one panel at a time** — inactive panels are *absent
from the DOM*, not hidden, so there is no cross-panel focus trap and no tabbable hidden content. The
**Energy tab** `splice`-inserts at **index 2** (after Climate=0, Charging=1) only when
`!energy.hide && hasEnergySite(_energy)`, without reflowing the user out of their current panel.
`default_panel` seats the first tab and the render **falls back to `charging`** if it names an
absent/hidden one.

## 5. Entity Resolution (the `data/` layer)

Resolution lives under **`src/data/`** (the catalog seed is in `src/const.ts`). It runs **once** in
the parent (identity-memoised) and the resolved map is passed down; children never re-resolve.

### 5.1 The catalog — `DEFAULT_ENTITIES` (`const.ts`)
An `as const` object mapping **84 stable function-name keys** (e.g. `battery_level`, `odometer`,
`time_to_full_charge`, `tire_fl`, `seat_rl`, `wake`, `boombox`) to the exact live entity IDs of a
reference "Garage Model Y" install. `EntityKey = keyof typeof DEFAULT_ENTITIES`, and the object is
`satisfies Record<VehicleKey, string>` — a typecheck tripwire that keeps the value table from drifting
from the registry vocabulary. There is **no `DEFAULT_IMAGE`** — the old `/local/model_y.png` fallback
was removed so a fresh install never 404s; the zero-config default hero is the bundled inline-SVG generic EV.

> ⚠️ **Tesla Fleet IDs are NOT uniformly prefixed.** ~30 of the keys point at *bare* device
> entities — `sensor.odometer`, `cover.sunroof`, `sensor.shift_state`, all tire pressure sensors,
> rear seat heaters, `sensor.speed`, `sensor.power`, climate cabin-overheat, … The rest carry the
> `garage_model_y_` device prefix. Blindly prefixing the bare ones yields `unavailable`.
> `DEFAULT_ENTITIES` encodes the exact mix.

### 5.2 Resolution flow (`data/resolve.ts`, `data/registry.ts`)
The function-key vocabulary (`ROLES`, `FUNCTION_KEYS`, `BUS_ORIENTATION`) is the single source of
truth in `data/registry.ts`. **`ROLES` = 7** (`vehicle` + the 6 energy roles), with **84 vehicle +
22 energy function-keys** (the value tables in `const.ts` and `energy.ts` are typecheck-bound back to
these names, so the three — registry, `const.ts`, and `energy.ts` — cannot diverge). `resolveEntities()` builds a complete id map for every key,
in order of precedence: explicit `config.entities[key]` override → the detected dialect's `DIALECT_ABSENT`
`''` sentinel for a key it doesn't expose (honest-unavailable, §5.5) → registry match by canonical identity
within the detected Tesla device (any prefix, `TESLA_PLATFORMS` set; the dialect's `DIALECT_ENTITY_ALIASES`
name substitutes for the fleet canonical when it diverges, §5.5) → direct prefix guess against live
`hass.states` → bare global id if present → bundled `DEFAULT_ENTITIES[key]` fallback. It **always returns
a value for every key**. Without `hass`, only overrides + defaults are used; it degrades gracefully when the
entity/device registry is absent (older HA, demo harness). `slugify` coerces non-string `name` (`String(name ?? '')`)
so a YAML `name: 2024` never crashes resolution or editor discovery (robustness requirement FR-24).

### 5.3 Energy resolution (`data/energy.ts`)
Energy/Wall-Connector entities are a **second, separate** resolution. They match by **stable
function-slug substring in the object-id** (e.g. any `*solar_power*` sensor via the `RULES` matcher) —
prefix-independent and tolerant of the integration's `_2` duplicate-entity suffix (**shortest matching
object-id wins**, since disabled duplicates carry no live state). `resolveEnergyEntities → _energy` runs
once in the parent; `hasEnergySite(_energy)` gates the Energy tab; `detectEnergySite` composes the two so
callers don't recombine and drift. `config.energy.entities` overrides always win; `config.energy.hide: true`
suppresses the panel even when a site is detected. Values are read via the NaN-safe
`numById`/`stateById`/`unitById`/`attrById` helpers (units read **live** — the Wall-Connector handle
temperature is **°F** on some installs; `attrById` is the one sanctioned direct-`hass.states` read in
`energy.ts`, for non-string attrs like `select` options).

### 5.4 Memoisation & propagation
`TeslaCard` memoises resolution on the identity of `hass.entities` / `hass.devices` / `_config`,
builds `_resolvedConfig = { ..._config, entities: resolveEntities(...) }`, and passes that resolved
`cfg` (+ `hass`) to every child. Children read ids via `helpers.ts` (`entityId`, `rawState`, …),
which read `config.entities[key]`.

### 5.5 Per-integration dialects (Epics 14–17)

Tesla surfaces through several HA integrations that name the same entity differently and expose different
subsets — Tesla Fleet, Teslemetry, Tessie, the community `tesla_custom`, and the legacy bare `tesla`
(Fleet-family). The `DIALECTS` table (`data/dialect.ts`) carries one adapter per key; **five keys total**
(`tesla_fleet` / `teslemetry` / `tessie` / `tesla_custom` / `tesla`).

- **Detection (`detectDialect(hass, config, scope?)`, Epics 14.2/16.2).** Returns a small typed
  `DialectReport` that *always* carries a working `integration` (plus the raw `candidates`). Precedence:
  explicit `config.integration` override → probe → ambiguity guard. `resolveEntities` feeds the **resolved
  vehicle device's** entity ids as the `scope` (empty ⇒ scope OMITTED = registry-wide probe), so a
  split-platform household (e.g. a `tesla_custom` car + a `tesla_fleet` Powerwall) probes to the car's
  single dialect. Picking that device is a guarded **four-tier** fallback in `detectVehicle` (`resolve.ts`):
  `config.device` → integration-override-steered → **vehicle-signature score** (`VEHICLE_SIGNATURES` suffix
  match — a car outscores a higher-entity-count Powerwall) → most-entities tie-break.
- **Name-alias resolution (Epic 14.1).** Two per-dialect tables drive `resolveEntities` (§5.2):
  `DIALECT_ENTITY_ALIASES` (canonical `EntityKey` → that dialect's slug-free `"domain.suffix"`) and
  `DIALECT_ABSENT` (keys a dialect doesn't expose → the `''` honest-unavailable sentinel). Consulted once
  per resolve via the detected dialect. **`tesla_fleet` and bare `tesla` have NO entry in either table** ⇒
  byte-identical fleet resolution, pinned by `src/fixtures/resolve-fleet-golden.json`. (The `tessie` table
  was corrected against upstream source in Story 16.1.)
- **The effective-dialect stamp (Epics 15.1 / 17.2).** `resolve.ts` exports `detectVehicleDialect(hass,
  config)` — the resolver's **effective** vehicle-scoped dialect *including the ambiguity collapse*,
  single-sourced with `resolveEntities`' own consult via the private `effectiveDialect`. `tesla-card.ts`
  `_resolve()` **stamps** it as `integration:` onto `_resolvedConfig` (in-memory only — never persisted, no
  `config-changed` dispatch), so children may treat `config.integration` as the settled dialect and
  `adapterFor` short-circuits on that override branch. The collapse itself is the pure
  `collapseDialect(report)` (ambiguous ⇒ `tesla_fleet`, else `report.integration` verbatim) — the codebase's
  **one** collapse; `adapterFor(hass, config)` dispatches `DIALECTS[collapseDialect(detectDialect(...))]`
  through the same export, so even an *unstamped* caller on an ambiguous registry deterministically gets the
  fleet adapter.
- **Charging classified end-to-end (Epics 15/16).** The Hero (`_chargeVisual`) and `panel-charging.ts`
  both classify via `adapterFor(this.hass, this.config).normalizeChargingState(...)` fed to the shared
  7→3 `classifyChargeState` (`charging`/`plugged`/`parked`). `tesla_custom` charging is a **boolean
  capability** — `on → charging`, `off → 'unknown'` — so the Hero's physical `charge_cable` corroboration
  resolves the tie (off + cabled → plugged, off + uncabled → parked). The charging panel renders the
  canonical `STRINGS.status` word whenever the dialect's override **covers** the raw value
  (`chargingOverrideCovers`, single-sourced with the Hero — Story 16.1).

> 🔑 **Rule:** never hard-code a Tesla entity ID (vehicle **or** energy) in a component. Resolve by
> stable function-name through the parent's resolved config so the card works across installs/prefixes
> **and dialects**.

## 6. Live Energy Flow (the `flow/` layer)

The energy-flow engine is the card's most-verified subsystem. **One sign/unit convention, declared
exactly once** in `flow/balance.ts`: **kW everywhere; battery + = charging; grid + = import**; an
edge's `from→to` is its positive-flow sense. Every surface that infers charge direction consumes this
one authority — there is **no private copy**.

- **Two sign conventions, never conflated.** *Raw* sensor signs (what the panels/ecosystem cards read):
  `battery_power − = charging`, `grid_power + = import`, others ≥ 0. *Canonical* signs (what the
  FlowModel/balance use): `battery + = charging`, `grid + = import`. The raw→canonical flip happens
  **once**, at the `data/dialect` boundary (`DEFAULT_FLIP = ['powerwall']`), *before* the model sees
  anything. `BUS_ORIENTATION` (role → ±1) is a *third*, separate concern — it maps a canonical reading
  onto bus topology (grid `+1`: import injects into the bus; powerwall `−1`: charging draws from it),
  so it can't be read off the sign convention alone. Each lives in exactly one place.
- **`computeBalance` (`flow/balance.ts`)** is the sole balance authority — per-node net + per-edge
  balance, **role-generic (no per-node-type branches)**. It is the #1 verification target: property tests
  assert sign-convention correctness **and** conservation (per-node sum-to-zero; `net[BUS] ≈ −residual`).
  `IDLE_KW` — the direction **deadband**, defined in `flow/model.ts` (aliased `DEADBAND` in `binding.ts`) —
  is **0.05 kW**: a sub-deadband or NaN/`unavailable` read makes an edge **quiescent** (present,
  `direction:'none'`), never broken — sensor jitter never animates as flow. `balance.ts`'s own
  **`EPSILON_KW` = 0.05** is a *separate* concern — the conservation-check tolerance (`|residual| ≤
  EPSILON_KW`): **same value, different meaning; never conflate the two.**
- **`bindFlowModel(hass, config, opts, hide)` (`flow/binding.ts`)** auto-detects the **six** energy roles
  (`ENERGY_ROLES`) via the `data/energy` resolver; `POWER_KEY` maps each role to its single `*_power`
  sensor; `provenance ∈ {measured|inferred|quiescent}` tags each edge. The `flowInputsFrom` seam is
  where node **hide** and **multi-instance** are applied — never a render-only filter: hide is a
  dedicated `readonly Role[]` param forcing `kW:undefined` via the shared `absentInput` (so
  **hidden == absent** structurally); multi-instance flat-maps role→N inputs (instance override wins).
- **`FlowRenderer { update(model) }` (`flow/renderer.ts`)** is the renderer seam. Its sole live
  implementation is `SceneBusRenderer` (`flow/scene-bus.ts`) — node-id → live `getBoundingClientRect()`
  anchors (screen px) — consuming **one** kW→visual helper (`edgeVisual`: `width = 1.6 + |kW|·0.55`,
  `durSec = max(0.5, 1.7 − |kW|·0.16)`; `NODE_COLOR`/`NODE_ICON`). A test deep-equals its per-edge
  visuals against the shared `edgeVisuals` (the renderer never forks the math). Because `NODE_COLOR` /
  `NODE_ICON` / the `edgeVisual` formula are **value-level exports the typecheck can't see**, their value
  pins live in the dedicated `flow/renderer.test.ts` (since `e478cbc`); `flow/scene-bus.test.ts` keeps only
  the R1 integration proof. The `FlowRenderer` interface seam remains (AR-7, revised to **one renderer**: a single live
  renderer behind one model); the Hero overlay renderer (`HeroSvgRenderer`/`flow/hero-svg.ts`) was
  removed in Story 12.1, so `SceneBusRenderer` is now the sole implementation.
- **Colour-blind safe:** every edge carries a node-anchored label + kW magnitude (source is never
  hue-only). **Reduced-motion → a legible static read** (arrowheads + kW labels, no animation).

The recognizable flow states (`src/fixtures/flow-*.json` — grid import / export, solar surplus,
charging vs plugged-idle, storm-watch/islanding, vampire drain) are each a **committed fixture + a
FlowModel assertion** (Epic 9 adds `energy-generator.json`).

> 🔗 **Composed-view authority split:** the discrete `charging_status` entity owns the Hero halo
> (`hero._chargeVisual()`); the FlowModel owns the wall-connector edge magnitude/direction — the WC
> edge **is** the car-charging edge (the vehicle is **not** a flow node). A test asserts halo and edge agree.

### 6.1 The frozen authority — AR-6

`flow/` is **7 non-test files** (`model`, `balance`, `binding`, `renderer`, `scene-bus`, `my-home`,
**`instances`** added in Story 9.7; `hero-svg` removed in Story 12.1). The freeze is **not** a blanket
lock on all of them: Epic 9's new role and multi-instance support deliberately edited `binding.ts` /
`renderer.ts` / `hero-svg.ts` / `scene-bus.ts` and added one additive seam to `model.ts`. The precise
invariant is **AR-6** — requirement-side name **FR-33** (verified against git this scan):

- **`flow/balance.ts` has ZERO production diff since Story 4.1** (a single commit in its history) — the
  true frozen compute + sign authority; the freeze has now held across **15 consecutive epics** (through
  Epic 17). The kW→visual math in `renderer.ts` (`edgeVisual`) is likewise math-unchanged.
- **`flow/model.ts` carries only the additive multi-instance identity seam** — `git diff` of Story 4.2 →
  9.7 shows the *only* logic change is adding optional `FlowInput.id`, then `const id = input.id ?? input.role`,
  keying `FlowNode.id`/`FlowEdge.from` off `id`. The balance math `BUS_ORIENTATION[input.role] * input.kW`
  is untouched, so a single instance (`input.id === undefined`) yields **byte-identical** pre-9.7 output.
- **A new energy node/role is a registry + component-metadata edit — NEVER a `balance.ts`/`buildFlowModel`-math
  edit.** Adding to `ROLES` makes the compiler enumerate exactly the role-keyed tables
  (`FUNCTION_KEYS`/`BUS_ORIENTATION`/`POWER_KEY`/`NODE_COLOR`/`NODE_ICON`) to fill — the
  typecheck cascade is the witness that balance/model math is never named. `generator` (Story 9.14) is the
  live witness (§12.5).

> The freeze is enforced **behaviourally, not by a golden-file diff gate**: `flow/balance.test.ts` proves a
> hidden role's `net` is node-for-node identical to an absent one; `audit-r6-suite` asserts the engine
> carries exactly `ENERGY_ROLES`, the vehicle is never a flow node, and `computeBalance().net` is identical
> across install prefixes; the Scene e2e proves an *absent* generator leaves "today's exact Scene."

## 7. Vehicle Controls & Panels (Epic 5)

The control surface declares **two opposite feedback contracts, once each**:

- **Optimistic-then-reconcile** (`quick-actions.ts` is canonical; generalized to boolean/number/enum in
  `panel-climate.ts`, reused in `panel-media.ts` and `powerwall.ts`): the UI flips optimistically, then
  **reconciles by re-deriving from the live predicate** every `hass` tick (never a tap-time snapshot). The
  sighted render follows the optimistic value but the **SR name** (`aria-pressed`/`aria-label` via
  `srState`) follows the settled real state. The honesty fence is the exported **`RECONCILE_TIMEOUT_MS`
  (10 s)** — a bounded `setTimeout`, cleared on reconcile and `disconnectedCallback`. *Import it; never
  redefine the magic number* (`quick-actions` exports the one constant; `panel-climate`,
  `panel-media`, and `powerwall` import it).
- **Fire-and-forget** (`commands.ts`): a one-shot `button.press`, no optimistic flip / reconcile /
  timer. Read-only panels (tires/location), physical-closure actions, and media prev/next are also
  deliberately non-optimistic (an optimistic flip would be false certainty).

**The three glanceable charge states** are a *dialect-aware* derived read, not a raw value:
`hero._chargeVisual()` feeds `adapterFor(this.hass, this.config).normalizeChargingState(charging_status)`
(the dialect adapter, §5.5) into the **shared `classifyChargeState`** (`data/dialect`) → **`charging` /
`plugged` / `parked`** (`unknown`/absent → **Parked** unless the physical `charge_cable` sensor corroborates
**plugged** — never a fabricated state). `panel-charging.ts` classifies through the *same* two functions, so
Hero and panel can't disagree (Story 16.1). `normalizeChargingState` itself returns the finer
`charging|starting|stopped|complete|disconnected|no_power|unknown`; `classifyChargeState` is the 7→3 collapse.
**Asleep outranks everything** (Story 17.1): the shared `isAsleep` predicate (`helpers.ts`, consulted by both
Hero and panel) forces the visual to `parked` — a sleeping car's cached "Charging" never claims a live
connection — outranking even the `unavailable`→"Idle" short-circuit. Live kW is a direct NaN-safe
`charger_power` read — the Hero holds **no copy** of the flow sign convention.

**Stateless `button` domain.** `button` entities have no meaningful state (`'unknown'` until first
press, reset every HA restart). Commands degrade on **missing only** (`isMissing` = `undefined`/`'unavailable'`),
**never** `isUnavailable`/`UNAVAILABLE_STATES` (which include `'unknown'` and would disable every never-pressed
command, including wake, on a fresh install).

**Wake citizenship is a CI-blocking invariant** (`data/wake.ts`): `observedWakeState`/`canWake` never
issue `button.press('wake')` under an observed `online`/`waking` state. Classification order is load-bearing
(`online → waking → asleep → unknown`); `waking` is checked **before** `asleep` so the post-press tick
rate-limits frantic taps. The `online` signal requires a *fresh* `'on'` (a stale `'on'` degrades to
`unknown`), so an explicit user wake of a genuinely asleep car is never blocked. The gate holds no clock
(recompute-on-render); per-instance `config.wake_cooldown` (default 60 s) rate-limits implicit wakes.

**Honest freshness is multi-dimensional** — value, tone, AND computed baseline can each independently
lie ("never a false closed" is the one unforgivable copy error). `panel-closures` refuses "All closed"
unless every closure is `available && closed`; even a permitted stale "All closed" drops its confident
green to a `dim` tone. Stale/asleep surfaces show last-known + an "updated N ago" stamp via
`keyAgeHint`/`formatAgeHint` and `referenceNow(hass)` (the max server stamp, computed once per render) —
**never** `Date.now()`. (Media is the inverse: no stamp — a stale player degrades to a calm empty state.)
Peer-derived thresholds (tires) use the **fresh-corner subset only** so a stale reading can't inflate the
baseline. Cover/lock reads go through `normalizeCoverState`/`normalizeLockState` (`data/dialect`), never
inline `=== 'open'`/`'locked'`.

Shared primitives are built where first consumed, extracted where first shared: `tc-slider`
(`slider.ts`, commits `value-changed` on pointer/key **release only** — never per drag tick, to respect
the metered Fleet API) and `statTile` (returns `nothing` when its entity is missing — fewer tiles, not a
wall of dashes) land in charging and are reused across panels.

## 9. Design System (tokens & styles)

Defined once in **`src/styles.ts`**: `tokens` (a `:host { … }` block of `--tc-*` custom properties set
on the `tesla-card` host) and `sharedStyles` (reusable class rules + the locked a11y keyframe corpus
`{tc-pulse, tc-shimmer}`). CSS custom properties inherit across shadow DOM, so **children import only
`sharedStyles` + their own `css`** and read tokens via `var(--tc-…)`.

> 🔒 **Two hard token gates (`styles.test.ts` + `token-defined.mjs`):**
> 1. Every `var(--tc-*)` consumption MUST carry the DESIGN.md value as a **fallback** (`var(--tc-blue, #38bdf8)`).
> 2. **AND** the token must be a **real token** declared in `styles.ts` (or set locally) — the
>    `token-defined` gate (Epic 9) closes the Epic-8 blind spot where a fallback *existed* but the token
>    did not (two shipped bugs). The fallback gate proves a fallback exists; the token-defined gate
>    proves the token is real. **Both must pass.** Also: the "exactly one 180deg" gate is comment-blind —
>    write `180°` in prose, not `180deg`.

`styles.ts` also exports machine-checkable contract maps: **`ACCENT_SEMANTICS` (8 accents → meaning:
blue/green/amber/red/purple/orange/teal + `copper` `#c2855b` = generator/fuel)**,
`INTERACTION_PRIMITIVES`/`INTERACTION_BANS`, `BREAKPOINTS` (`{compact:540, full:760}` — a **build-time TS
constant**, tree-shaken, since neither CSS `@media` nor `@container` can read custom properties; a gate

single-sourced via `unsafeCSS` for both the `:host([theme='light'])` override block and the editor's live
preview). The pattern: an exported `as const` map is the machine-checkable half; comments are the human half.

The recolorable hero composites layers (`color` base → inside `mask`: a `--tc-paint` fill → `shade`
×multiply → optional `highlight` ×screen, `isolation:isolate`); `resolvePaint` (`paint.ts`) accepts three

token/primitive/helper tables, and [`recolorable-body.md`](./recolorable-body.md) /

## 10. Lovelace Card Contract

## 11. Build Pipeline

```mermaid

    src["src/tesla-card.ts"] -->|"Rollup — rollup.config.mjs"| dist["dist/tesla-card.js"]
```

- `format: 'es'`, `inlineDynamicImports: true` (single file, lazy editor inlined), sourcemap in watch only.

  `../dist/tesla-card.js` → `src/tesla-card.ts`); the shipped bundle is **Rollup**, deliberately decoupled.

**Verification gates** (all merge-blocking): `npm run typecheck` (strict, no-emit), `npm run test`
(Vitest — counts in §2), `npm run lint` (the **8-gate** chain `no-bare-hass-states → no-cycle
→ trade-dress-denylist → import-allowlist → no-network-egress → version-sync → token-defined →
no-planning-artifacts`), and `npm run build`. A separate **test-inventory census**
(`tests/test-census.json` — the authoritative unit/e2e counts + e2e-spec-file list, enforced by
`scripts/lint/test-census.mjs`; regenerate with `npm run test:census -- --write`) catches silent
test-count drift — it runs in `test.yml` and `ci:local`, **not** in the 8-gate `npm run lint`. CI splits
these across `validate.yml` (HACS validation + build-asserts-bundle + the 8 structural gates), `test.yml`
(typecheck, Vitest unit, Playwright e2e, the census gate, weekly burn-in), and `release.yml` (version-sync
`--release-tag` → build → attach the single bundle).
See [`ci.md`](./ci.md) and [`development-guide.md`](./development-guide.md).

**Release-time checks (deliberately outside `npm run lint`):**

- **`scripts/lint/pre-flip-guard.sh`** — the human-run gate before the public-visibility flip /
  release cut. The `no-planning-artifacts` lint gate is HEAD-only and name-only, but flipping a repo
  public exposes all history and file content, so the guard layers four checks: the tracked-tree name
  gate (L1), a content fingerprint over tracked files (L2), planning-artifact names anywhere in git
  history (L3), and a secret/credential smell-test over tree + history (L4). Exit 0 = clear to flip.
- **The NFR-1 profiler ladder** — `npm run profile:nfr1` (`scripts/profiler/fps-probe.sh`) drives the
  composed Scene in the demo harness against
  [`profiler-checklist-nfr1.md`](./profiler-checklist-nfr1.md). **NFR-1** (the composed Scene sustains
  ~60 fps) last validated 2026-06-21: instrument-driven 120 fps / zero jank under a validated 6× CPU
  throttle.
- **`npm run ci:local`** mirrors the exact CI chain locally (the `validate.yml` build job + the
  `test.yml` lint/test jobs, with `CI=1` Playwright parity) before a push.

**The demo harness (`demo/index.html`)** is the shared zero-HA host for Playwright e2e, the NFR-1
profiler, README screenshots, and manual dev. `npm run serve:demo` serves the repo root on `:4173`
(Playwright's `webServer` runs `npm run build && npm run serve:demo` itself); the page imports
`../dist/tesla-card.js` and feeds it fixture-driven `hass` states. URL params select the surface:

does, via `getConfigElement`), `&editortype=my-home` (the Scene-aware editor, Story 10.1). It ships
in-repo but is not part of the bundle.

### 11.1 Security, privacy & egress posture

The card is **egress-free by construction and by gate**: its only I/O is the injected `hass` object —
`hass.states` reads (in-memory page state, not a network call), `hass.callService` writes, and the one
recorder-history read over `hass.callWS` (HA's own authenticated socket). It opens **no socket of its
own**: no `fetch`/XHR/WebSocket, no telemetry or analytics, no CDN scripts/fonts/tiles (the location
map renders from GPS *attributes*; HA's map stack fetches its own tiles outside the card's bundle).
Enforcement is two-layered — the merge-blocking `no-network-egress` gate plus human review. See
[`privacy.md`](./privacy.md) for the full data-flow statement, and
[`audit-r6-suite.md`](./audit-r6-suite.md) / [`audit-r6-vehicle-card.md`](./audit-r6-vehicle-card.md)
for the standing R6 audit-checkpoint reports behind §6.1's freeze claims.

## 12. Enhanced Configuration — the no-YAML editor & customization (Epic 9 / CAP-5)

Epic 9 is the **1.0.0 net-new capability**: a complete no-YAML setup / theming / customization
experience, "My Home" Scene customization, and a new energy node **type**. **Every Epic-9 config key is
additive and zero-diff-when-absent** — omitted ⇒ today's behaviour byte-for-byte.

### 12.1 The GUI editor (`src/editor.ts`, ~2,656 LOC, `@customElement('tesla-card-editor')`)

- **5-step skippable wizard** `detect → confirm → appearance → tune → finish` (`WIZARD_STEPS`), plus a
  **normal form** (discovery summary → appearance → tune → name/image → Scene-node customization → "Run
  guided setup" re-entry). One shared component renders each surface in **both homes**; `_wizardActive`
  picks the home (an explicit override, else `setup_complete`, else "is this a bare stub?").
- **Persistence is per-step partial-config-to-Lovelace, NEVER `localStorage`/component state.** Resume
  marker = additive `setup_complete?` (`true` = done/normal-form; `false` = in-progress/resume; absent =
  auto-open only for a bare stub). Resume is derived in `willUpdate` (guarded by a one-shot `_resumed`
  flag so HA echoing your own `config-changed` doesn't reset within-session nav), never `render()`. The
  **resume promise (D8):** survives editor reopen, browser refresh, and cross-device — each step persists a
  forward-compatible (R9) config; the auto-detected config is written at/after **Step 1 (Detect)** so a
  hard refresh before first save can't discard a brand-new card.
- **`_emit` vs `_patch` discipline:** `_emit(config)` = whole-config **REPLACE** + `config-changed` —
  use whenever a key may be **deleted** (a merge can't delete). `_patch(patch)` = shallow **merge** —
  only for pure SETs (in practice, narrowly: the `setup_complete:false` Detect write). **Every reset/clear/default
  DELETES the key** (a removed key, never a blanked value) and prunes emptied containers by
  `Object.keys(x).length > 0` (preserving unknown/future keys). A `safeClone` guard normalizes a malformed
  hand-edited container to `{}` so it never throws.

### 12.2 Editor `hass.states` boundary (Story 9.10 / decision D7)

The Epic-7 "editor reads no `hass.states`" rule is deliberately **relaxed for the editor only**. The
editor MAY read its **own** `hass` for discovery: registry maps (`hass.config.components → hass.devices
→ hass.entities`) for **presence**, and `hass.states[id]` **only to annotate liveness**. Guardrails:
public `hass` surface only; **NEVER `import` an HA-frontend `src/data/*` helper** (that breaks the
bundle); key→entity resolution still flows through the card's unchanged `data/` resolvers. Discovery is
re-derived fresh per editor open (never persisted) — only resolved entity-IDs persist. The exception is
a **self-invalidating** `'src/editor.ts'` baseline entry in `no-bare-hass-states.mjs` (the gate FAILS if
`editor.ts` ever stops containing a bare read — the boundary ratchets shut, never reopens). The four-state
liveness vocabulary, computed by `_liveness` off `hass.entities` (the registry) + `hass.states`:
**✓ online · ⚠ unavailable · ⚠ no data yet (`unknown`) · — not found** (✓ attests *reachable*, not *awake*;
the "not found" word sits at `--tc-text-dim` for AA, never `--tc-text-mute`).

### 12.3 Customization surfaces

- **Per-entity overrides (9.11)** — `<ha-selector>` used directly in the Lit template (a globally-registered
  HA custom element, **no JS import** → `import-allowlist` satisfied by construction; bind the bare config
  to `.selector`, read `ev.detail.value`). A single `OVERRIDE_TARGET` map keys both discovery (read) and
  the picker (write): vehicle → `config.entities[key]`, energy → `config.energy.entities[key]`.
  Accordion-in-place; a present row gets a per-role permissive filter list; a `— not found` row opens an
  **unfiltered** picker ("map-a-miss"); an honest dead-pick is saved regardless with a polite ⚠ announce;
  reset-to-auto **deletes** the key (shown only when an override is set).
- **Appearance & theming (9.12)** — paint swatches carry + write the **curated HEX** from `PAINT_PRESETS`,
  not the bare key (`resolvePaint`'s literal-string branch returns a CSS keyword verbatim *before* presets —
  writing `'blue'` rendered a garish primary `#0000ff`; the epic's one HIGH bug). Theme override is
  **card-only** (`appearance.theme?: 'light'|'dark'`, Default = absent): a host `theme` attribute (reflected in
  `tesla-card.ts:updated()`) → a `:host([theme='light'])` token-override block re-resolves the colour tokens,
  single-sourced from `LIGHT_TOKENS` (never an HA global theme var). The hex field is a **`{ text: {} }`**
  selector (NOT `color_rgb`/`type:color`) written verbatim — never clamped/validated/substituted. Paint-swatch
  grid + Default/Light/Dark control are **hand-rolled `radiogroup`s** (roving tabindex). Full-card **live preview**
  reuses the real `carView`, coalesced per pick.
- **Tune step + version floor (9.13 / D8)** — editor floor = runtime floor = **`2024.4.0`, no raise**
  (every chosen widget predates it; the entity-selector `filter:` form shipped HA 2023.3). The README
  caveat is **API-stability** (`ha-selector` is HA-internal, non-SemVer), **not** version exclusion.
  `TiresConfig.units?: 'psi'|'bar'` is **display-only** (thresholds stay stored in the sensor's native unit);
  `_convertPressure` pivots through bar (psi/bar/**kPa**, `PSI_PER_BAR = 14.5038`), and `_tireNativeUnit()`
  sniffs the sensor unit — **the kPa branch mirrors `panel-tires`** (a post-Epic-9 review-hardening
  kPa-parity fix, merged to `main`). Tune toggles/dropdowns are per-card globals — never instance-suffixed.

### 12.4 "My Home" node customization (9.1–9.4, 9.7, 9.8, 9.15) — `energy.nodes`

- **Hide at the MODEL seam, reorder/promote at the RENDER seam.** Hide forces `kW:undefined` at
  `flowInputsFrom` (a shared `absentInput(role)` makes "hidden == absent" structural) so `buildFlowModel`
  emits `present:false` — filtering in `render()` would desync the SVG bus from the grid (forbidden).
  Reorder is the grid-row packing (`_orderedRows`); the bus follows because `gatewaySegments` sorts taps
  by spatial anchor centre-x. Precedence: **hide wins over order.** The keyspace is `Role` (incl. vehicle),
  never `EnergyRole`.
- **Cross-row promotion (9.15)** — `energy.nodes.rows` relaxes the within-row default. **Bus sign ≠
  row:** a node's bus contribution is its own measured net × `BUS_ORIENTATION[role]`, row-invariant by
  construction (balance never reads the rendered row). A single `rowOf(role, cfg)` classifier drives both
  render and geometry so they can't disagree.
- **Multi-instance / per-instance identity (9.7) — `flow/instances.ts`.** Node identity is computed
  `instanceId = count <= 1 ? role : ${role}:${i+1}` — **bare for single = zero-diff**, `:1`/`:2` only
  for genuine duplicates. The id is threaded through every seam that assumed `id===role` (`FlowNode.id`,
  the anchor map, the `edgeById` edge lookup, `scene-bus` chip lookup, focus coupling). The `instances` schema is a
  descriptor list `Partial<Record<Role, InstanceSpec[]>>` (`{ title?, entities?, config? }`); array
  length = the instance count; `config?` is consumed only for the `vehicle` role (a 2nd car's embedded
  card). N instances are consumed at the binding seam (`flowInputsFrom` flat-maps) — `balance.ts` is
  **not** edited (it already aggregates by node id). **WRAP overflow (design decision D15 — overflow
  wraps; never scrolls or clamps):** a row band > `WRAP_MAX_PER_ROW
  = 3` drops extras to a 2nd sub-row offset **`230px`** (`= (380 + 80)/2`, **not** the mock's 182px) into
  the bus channels (`SAFE_BAND_MAX = 6`); the overflow sub-row uses CSS `order:-1` visually while DOM/Tab
  order stays primary→overflow L→R; disambiguate duplicated roles by card **title**, never a numeric `:n` badge.
- **Detected-but-hidden advisory** rides the rendered card (`my-home.ts`), not the editor — an
  "opposite-of-hide" probe with `role="status"`/`aria-live="polite"` (never red, never auto-un-hides);
  global toggle `notify_hidden_detected?` (default-on).

### 12.5 Generator — a new node TYPE (Story 9.14)

The witness for AR-6 (§6.1): adding `generator` to `ROLES` made the compiler enumerate exactly the
role-keyed tables to fill — `FUNCTION_KEYS.generator = ['generator_power']`, **`BUS_ORIENTATION.generator
= 1`** (source sign, like solar), `EnergyEntities.generator_power?`, `POWER_KEY`, `NODE_COLOR`,
`NODE_ICON` (`mdiGeneratorStationary`), editor `NODE_LABELS` — and **never** `balance.ts`/
`buildFlowModel` math. Because `ENERGY_ROLES = Object.keys(POWER_KEY)` and `SCENE_NODES = ENERGY_ROLES`,
the generator auto-flows through binding→model→balance→ribbon→bus by construction. **Copper `#c2855b`**
is the 8th `ACCENT_SEMANTICS` accent (with `--tc-copper` a real token — both halves the token-defined gate
needs); `tc-generator` is modeled on the *simple* ecosystem mold (`grid.ts`/`home.ts`, no vignette/charts),
and is the 8th `window.customCards` picker entry.

## 13. Key Constraints & Gotchas

- **`useDefineForClassFields: false`** is load-bearing (see §2). Never "fix" it.
- **No hard-coded Tesla entity IDs** — resolve by function-name (see §5). Energy ids auto-detect too.
- **Dialects resolve + collapse in ONE place** — `data/dialect.ts` owns detection, the alias/ABSENT
  tables, and the single `collapseDialect`; the parent stamps the *effective* vehicle-scoped dialect onto
  `config.integration` (in-memory, never persisted) so children **and** `adapterFor` agree by construction
  (§5.5). Never re-derive a dialect inside a component.
- **One sign convention per layer, never copied** — canonical (`flow/balance.ts`), RAW in ecosystem
  cards/panels, and Scene ribbon/bus VIEWs of `computeBalance().net`; the raw→canonical flip happens once,
  at `data/dialect` (§6, §8).
- **`data/ ← flow/ ← components/`** direction is gate-enforced; only `src/data/` reads `hass.states` at
  runtime (only `data/history.ts` reaches recorder history, via `callWS`); the **editor** is the D7
  exception (§3, §12).
- **Two token gates** — every `var(--tc-*)` carries a fallback **and** names a real token (§9).
- **The frozen authority is `flow/balance.ts` (AR-6), not "all six flow files"** — `flow/` is now 7 files;
  a new node/role is registry + component metadata; `model.ts` carries only the `id ?? role` seam (§6.1).
- **Reuse the Epic-5 feedback machines** — import `RECONCILE_TIMEOUT_MS`, never redefine it.
- **Keep the Lovelace contract** (`setConfig`/`getCardSize`/`getStubConfig`/`getConfigElement`); honour
  the forward-compat spread.
- **Every Epic-9+ config key is additive and zero-diff-when-absent**; in the editor, reset/clear/default
  **deletes** the key via whole-config `_emit` (a merge can't delete) — never a blanked value (§12.1).
- **`variant: 'compact'` is presentation-only** — it never force-hides sections; element-relative
  responsive collapse is via **container queries** (`container-type: inline-size` + `@container`, D-CQ-1),
  with the `:host([compact])` host attribute now a redundant backup — never a viewport `@media` (§8.1).
- **Never commit `dist/`** — gitignored, built in CI. **Version sync:** `package.json` `version` ↔
  `src/const.ts` `CARD_VERSION` (`1.0.0`) ↔ git tag must all match (the `version-sync` gate).
- **Standalone public repo** — commit card changes *inside* `tesla-card/`. BMAD planning artifacts live
  in the sibling private `tesla-card-planning/` repo (gate-blocked here).
- **No vehicle artwork ships** — the bundled default is a generic EV silhouette; bring your own layer
  packs (HA-served from `/local/tesla-card/<model>/`, never committed). See
  [`asset-packs.md`](./asset-packs.md) / [`trade-dress.md`](./trade-dress.md).
- **Sanctioned hard-coded colours** (the only departures from `--tc-*` tokens): the `panel-location`
  `135deg` map gradient (`#1b2533`/`#0f1620`), the `tc-slider` `#fff` thumb, the Scene `GATEWAY_STROKE`
  `#cfe2ff`, the generic-EV SVG art, and the aperture skins. All others are tokens.
