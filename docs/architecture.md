## 1. Executive Summary

`tesla-card` is a Tesla-app-style custom [Lovelace](https://www.home-assistant.io/dashboards/)
card for Home Assistant, built in **TypeScript (strict) + [Lit](https://lit.dev) 3**. It renders a
single vehicle as a centred **living hero** (recolorable inline-SVG car with charge/aperture state)
over a **tab bar** of feature panels (charging, climate, energy, closures, tires, location, media),
plus quick-action toggles and command buttons. Beyond the vehicle card it also ships a set of
standalone **ecosystem cards** (solar / powerwall / grid / home / wall-connector / **generator**) and a
composed **"My Home" energy Scene**, all from the same bundle.

Three capability arcs define the current shape:

- **Live, honest energy flow.** One sign/unit convention and a single balance authority drive the
  Scene's Gateway bus, while the Hero's charge halo reads the discrete `charging_status` entity ŌĆö a test
  asserts the two agree, so the visuals never disagree with the numbers or animate a false flow (Epics 4/6/8).
- **No-YAML Enhanced Configuration (Epic 9 / the 1.0.0 net-new).** An HA-native GUI editor
  delivers a guided first-run wizard + a normal form for theming, per-entity overrides, and "My Home"
  Scene customization (hide / reorder / cross-row promote / duplicate nodes), plus a brand-new
  `generator` node **type** ŌĆö all additive and zero-diff-when-absent (┬¦12).
- **Post-1.0 polish (Epics 10 & 11, off the critical path).** Epic 10 adds a Scene-aware mode of the
  same editor for `custom:tc-my-home` (┬¦12.7); Epic 11 adds render-polish for the composed Scene + its
  embedded vehicle cell (┬¦8.1).

It ships as a **single bundled ES module** (`dist/tesla-card.js`) consumed by Home Assistant as a
Lovelace resource and distributed via [HACS](https://hacs.xyz/). There is **no application server,
no API layer, and no datastore** ŌĆö all state comes from the HA `hass` object injected by the
dashboard at runtime, every service call goes back through `hass.callService`, and the one historical
read rides `hass.callWS` (HA's own authenticated WebSocket ŌĆö never a socket of the card's own).

The card's robustness rests on **entity resolution by stable function-name** ŌĆö not hard-coded IDs ŌĆö so it
works across installs, device renames, and integration prefixes (┬¦5).

## 3. Layered Architecture ŌĆö `data/ ŌåÉ flow/ ŌåÉ components/`

The source is organized into three layers with a **one-way dependency direction**, enforced by the
`no-cycle` and `no-bare-hass-states` lint gates:

```mermaid
%%{init: {"flowchart": {"wrappingWidth": 500}}}%%
flowchart TB
    hass[("hass.states")]
    data["<b>data/</b> ŌĆö freshness ┬Ę dialect ┬Ę energy ┬Ę history ┬Ę registry ┬Ę resolve ┬Ę slice ┬Ę wake<br/>pure read-model + normalization boundary (8 modules)"]
    flow["<b>flow/</b> ŌĆö balance ┬Ę model ┬Ę binding ┬Ę renderer ┬Ę scene-bus ┬Ę my-home ┬Ę instances<br/>pure energy-flow math (7 modules); imports data/, never components/"]
    components["<b>components/</b> ŌĆö tc-* + helpers<br/>hero / panels / quick-actions / commands / ecosystem / scene + car / weather / chart / node-hero<br/>Lit elements + render helpers; import flow/ and data/"]
    hass -->|"read by data/ ŌĆö the ONLY layer allowed to"| data
    data -->|"canonical, sign-normalized data"| flow
    flow -->|"FlowModel + visual derivations"| components
    classDef dataL stroke:#3987e5,stroke-width:3px
    classDef flowL stroke:#199e70,stroke-width:3px
    classDef compL stroke:#c98500,stroke-width:3px
    class data dataL
    class flow flowL
    class components compL
```

- **`src/data/`** is the data-access boundary (8 modules). It is the **only** subtree allowed to read
  `hass.states`/`.entities`/`.devices`. `data/freshness.ts` owns the universal read-model
  (`read`/`readKey`/`readRaw`, `referenceNow`, staleness classification, `isQuiescent`); `data/resolve.ts`
  + `data/energy.ts` resolve entities; `data/dialect.ts` quarantines per-integration spelling
  (`normalizeChargingState`/`normalizeCoverState`/`normalizeLockState`, the `DIALECTS` table, the rawŌåÆcanonical
  power-sign flip); `data/registry.ts` holds the role vocabulary; `data/wake.ts` the wake gate;
  `data/slice.ts` the reflow gate; `data/history.ts` reaches HA recorder history over
  `hass.callWS('history/history_during_period')` (**not** `hass.states`, **never** polled) so the boundary
  still holds. Every `data/` module imports only root leaves + sibling `data/` ŌĆö **zero upward imports**.
- **`src/flow/`** is pure energy-flow logic (7 modules). It imports `data/` (and root leaves) and **never**
  `components/`. `flow/balance.ts` owns the single sign/unit convention; `flow/instances.ts` (Epic 9)
  is the DOM-free per-instance identity helper.
- **`src/components/`** holds the Lit elements (`tc-*`) and pure render helpers (`car.ts`,
  `weather-vignette.ts`, `chart.ts`, `node-hero.ts`). It imports `flow/` and `data/`.
- **Root leaf modules** (`strings.ts`, `log.ts`, `const.ts`) import nothing upward, keeping the cycle
  check trivially green. All user-facing copy lives in `strings.ts`; all `console.*` routes through
  `log.ts` (a single neutral `[tesla-card]` prefix, no brand colour).

> **The editor is the one sanctioned boundary exception (D7):** `src/editor.ts` MAY read its **own**
> `hass` for discovery ŌĆö the card **runtime** never does. Guardrails and the self-invalidating
> `no-bare-hass-states` baseline are in ┬¦12.2.

**Element count.** The bundle defines **20** custom elements total: `tesla-card`,

in `window.customCards` ŌĆö `tesla-card` + `tc-solar` / `tc-powerwall` / `tc-grid` / `tc-home` /
`tc-wall-connector` / `tc-generator` / `tc-my-home`. **Editor registration timing differs by
artifact:** at the **source entry** (Vitest, Vite dev) the editor is **not** registered at load ŌĆö the
dynamic import is genuinely lazy, pinned by `tesla-card.contract.test.ts`
(`editorDefinedAtBundleLoad` must be `false`) ŌĆö but in the **shipped bundle**, `inlineDynamicImports`
hoists the editor module, so its registration executes at bundle evaluation (all 20 defines observed
when loading `dist/` in this scan). What stays lazy everywhere is editor *instantiation*: no
`tesla-card-editor` node exists until `getConfigElement()`. `car.ts`, `weather-vignette.ts`,
`chart.ts`, `node-hero.ts`, and `ecosystem-card.ts` register **no** element (render helpers / abstract
base class).

**Base class ŌĆö `TcBase` (`src/base.ts`).** Extends `LitElement` and supplies exactly two reactive
properties to every vehicle child:

```ts
@property({ attribute: false }) public hass?: HomeAssistant;
@property({ attribute: false }) public config!: TeslaCardConfig;   // the RESOLVED config
```

Children read `config` (not `_config`). The elements that don't extend `TcBase` directly are
**`tc-slider`** and **`tesla-card-editor`** (extend `LitElement` directly), and the **ecosystem cards** +
**`tc-my-home`**, which extend `EcosystemCard` / `LitElement` because they are standalone surfaces, not
vehicle children.

**Registration.** Children are pulled in by **side-effect imports** in `src/tesla-card.ts`
(`import './components/hero';` ŌĆ”) ŌĆö each imported module runs its own `@customElement(...)` on load.
`tc-slider` is side-effect-imported by its consumers (`panel-charging.ts`, `panel-media.ts`,
`powerwall.ts`), not by the parent. **`tc-my-home` independently side-effect-imports all six ecosystem
children** (incl. `tc-generator`) because the Scene can mount standalone ŌĆö leaning on the `tesla-card`
entry's imports would leave a present node rendering a blank element. Assert registration against the
**bundle entry** (`tesla-card.contract.test.ts`), not a convenient internal module.

**One panel in the DOM.** The parent renders **one panel at a time** ŌĆö inactive panels are *absent
from the DOM*, not hidden, so there is no cross-panel focus trap and no tabbable hidden content. The
**Energy tab** `splice`-inserts at **index 2** (after Climate=0, Charging=1) only when
`!energy.hide && hasEnergySite(_energy)`, without reflowing the user out of their current panel.
`default_panel` seats the first tab and the render **falls back to `charging`** if it names an
absent/hidden one.

## 5. Entity Resolution (the `data/` layer)

Resolution lives under **`src/data/`** (the catalog seed is in `src/const.ts`). It runs **once** in
the parent (identity-memoised) and the resolved map is passed down; children never re-resolve.

### 5.1 The catalog ŌĆö `DEFAULT_ENTITIES` (`const.ts`)
An `as const` object mapping **84 stable function-name keys** (e.g. `battery_level`, `odometer`,
`time_to_full_charge`, `tire_fl`, `seat_rl`, `wake`, `boombox`) to the exact live entity IDs of a
reference "Garage Model Y" install. `EntityKey = keyof typeof DEFAULT_ENTITIES`, and the object is
`satisfies Record<VehicleKey, string>` ŌĆö a typecheck tripwire that keeps the value table from drifting
from the registry vocabulary. There is **no `DEFAULT_IMAGE`** ŌĆö the old `/local/model_y.png` fallback
was removed so a fresh install never 404s; the zero-config default hero is the bundled inline-SVG generic EV.

> ŌÜĀ’ĖÅ **Tesla Fleet IDs are NOT uniformly prefixed.** ~30 of the keys point at *bare* device
> entities ŌĆö `sensor.odometer`, `cover.sunroof`, `sensor.shift_state`, all tire pressure sensors,
> rear seat heaters, `sensor.speed`, `sensor.power`, climate cabin-overheat, ŌĆ” The rest carry the
> `garage_model_y_` device prefix. Blindly prefixing the bare ones yields `unavailable`.
> `DEFAULT_ENTITIES` encodes the exact mix.

### 5.2 Resolution flow (`data/resolve.ts`, `data/registry.ts`)
The function-key vocabulary (`ROLES`, `FUNCTION_KEYS`, `BUS_ORIENTATION`) is the single source of
truth in `data/registry.ts`. **`ROLES` = 7** (`vehicle` + the 6 energy roles), with **84 vehicle +
22 energy function-keys** (the value tables in `const.ts` and `energy.ts` are typecheck-bound back to
these names, so the three ŌĆö registry, `const.ts`, and `energy.ts` ŌĆö cannot diverge). `resolveEntities()` builds a complete id map for every key,
in order of precedence: explicit `config.entities[key]` override ŌåÆ registry match by canonical identity
within the detected Tesla device (any prefix, `TESLA_PLATFORMS` set) ŌåÆ direct prefix guess against live
`hass.states` ŌåÆ bare global id if present ŌåÆ bundled `DEFAULT_ENTITIES[key]` fallback. It **always returns
a value for every key**. Without `hass`, only overrides + defaults are used; it degrades gracefully when the
entity/device registry is absent (older HA, demo harness). `slugify` coerces non-string `name` (`String(name ?? '')`)
so a YAML `name: 2024` never crashes resolution or editor discovery (robustness requirement FR-24).

### 5.3 Energy resolution (`data/energy.ts`)
Energy/Wall-Connector entities are a **second, separate** resolution. They match by **stable
function-slug substring in the object-id** (e.g. any `*solar_power*` sensor via the `RULES` matcher) ŌĆö
prefix-independent and tolerant of the integration's `_2` duplicate-entity suffix (**shortest matching
object-id wins**, since disabled duplicates carry no live state). `resolveEnergyEntities ŌåÆ _energy` runs
once in the parent; `hasEnergySite(_energy)` gates the Energy tab; `detectEnergySite` composes the two so
callers don't recombine and drift. `config.energy.entities` overrides always win; `config.energy.hide: true`
suppresses the panel even when a site is detected. Values are read via the NaN-safe
`numById`/`stateById`/`unitById`/`attrById` helpers (units read **live** ŌĆö the Wall-Connector handle
temperature is **┬░F** on some installs; `attrById` is the one sanctioned direct-`hass.states` read in
`energy.ts`, for non-string attrs like `select` options).

### 5.4 Memoisation & propagation
`TeslaCard` memoises resolution on the identity of `hass.entities` / `hass.devices` / `_config`,
builds `_resolvedConfig = { ..._config, entities: resolveEntities(...) }`, and passes that resolved
`cfg` (+ `hass`) to every child. Children read ids via `helpers.ts` (`entityId`, `rawState`, ŌĆ”),
which read `config.entities[key]`.

> ­¤öæ **Rule:** never hard-code a Tesla entity ID (vehicle **or** energy) in a component. Resolve by
> stable function-name through the parent's resolved config so the card works across installs/prefixes.

## 6. Live Energy Flow (the `flow/` layer)

The energy-flow engine is the card's most-verified subsystem. **One sign/unit convention, declared
exactly once** in `flow/balance.ts`: **kW everywhere; battery + = charging; grid + = import**; an
edge's `fromŌåÆto` is its positive-flow sense. Every surface that infers charge direction consumes this
one authority ŌĆö there is **no private copy**.

- **Two sign conventions, never conflated.** *Raw* sensor signs (what the panels/ecosystem cards read):
  `battery_power ŌłÆ = charging`, `grid_power + = import`, others Ōēź 0. *Canonical* signs (what the
  FlowModel/balance use): `battery + = charging`, `grid + = import`. The rawŌåÆcanonical flip happens
  **once**, at the `data/dialect` boundary (`DEFAULT_FLIP = ['powerwall']`), *before* the model sees
  anything. `BUS_ORIENTATION` (role ŌåÆ ┬▒1) is a *third*, separate concern ŌĆö it maps a canonical reading
  onto bus topology (grid `+1`: import injects into the bus; powerwall `ŌłÆ1`: charging draws from it),
  so it can't be read off the sign convention alone. Each lives in exactly one place.
- **`computeBalance` (`flow/balance.ts`)** is the sole balance authority ŌĆö per-node net + per-edge
  balance, **role-generic (no per-node-type branches)**. It is the #1 verification target: property tests
  assert sign-convention correctness **and** conservation (per-node sum-to-zero; `net[BUS] Ōēł ŌłÆresidual`).
  `IDLE_KW` = `EPSILON_KW` = `DEADBAND` = **0.05 kW** (one value, three named uses): a sub-deadband or
  NaN/`unavailable` read makes an edge **quiescent** (present, `direction:'none'`), never broken ŌĆö sensor
  jitter never animates as flow.
- **`bindFlowModel(hass, config, opts, hide)` (`flow/binding.ts`)** auto-detects the **six** energy roles
  (`ENERGY_ROLES`) via the `data/energy` resolver; `POWER_KEY` maps each role to its single `*_power`
  sensor; `provenance Ōłł {measured|inferred|quiescent}` tags each edge. The `flowInputsFrom` seam is
  where node **hide** and **multi-instance** are applied ŌĆö never a render-only filter: hide is a
  dedicated `readonly Role[]` param forcing `kW:undefined` via the shared `absentInput` (so
  **hidden == absent** structurally); multi-instance flat-maps roleŌåÆN inputs (instance override wins).
- **`FlowRenderer { update(model) }` (`flow/renderer.ts`)** is the renderer seam. Its sole live
  implementation is `SceneBusRenderer` (`flow/scene-bus.ts`) ŌĆö node-id ŌåÆ live `getBoundingClientRect()`
  anchors (screen px) ŌĆö consuming **one** kWŌåÆvisual helper (`edgeVisual`: `width = 1.6 + |kW|┬Ę0.55`,
  `durSec = max(0.5, 1.7 ŌłÆ |kW|┬Ę0.16)`; `NODE_COLOR`/`NODE_ICON`). A test deep-equals its per-edge
  visuals against the shared `edgeVisuals` (the renderer never forks the math). The `FlowRenderer` interface seam remains (AR-7: one interface behind one model); the Hero
  overlay renderer (`HeroSvgRenderer`/`flow/hero-svg.ts`) was removed in Story 12.1.
- **Colour-blind safe:** every edge carries a node-anchored label + kW magnitude (source is never
  hue-only). **Reduced-motion ŌåÆ a legible static read** (arrowheads + kW labels, no animation).

The recognizable flow states (`src/fixtures/flow-*.json` ŌĆö grid import / export, solar surplus,
charging vs plugged-idle, storm-watch/islanding, vampire drain) are each a **committed fixture + a
FlowModel assertion** (Epic 9 adds `energy-generator.json`).

> ­¤öŚ **Composed-view authority split:** the discrete `charging_status` entity owns the Hero halo
> (`hero._chargeVisual()`); the FlowModel owns the wall-connector edge magnitude/direction ŌĆö the WC
> edge **is** the car-charging edge (the vehicle is **not** a flow node). A test asserts halo and edge agree.

### 6.1 The frozen authority ŌĆö AR-6

`flow/` is **7 non-test files** (`model`, `balance`, `binding`, `renderer`, `scene-bus`, `my-home`,
**`instances`** added in Story 9.7; `hero-svg` removed in Story 12.1). The freeze is **not** a blanket
lock on all of them: Epic 9's new role and multi-instance support deliberately edited `binding.ts` /
`renderer.ts` / `hero-svg.ts` / `scene-bus.ts` and added one additive seam to `model.ts`. The precise
invariant is **AR-6** ŌĆö requirement-side name **FR-33** (verified against git this scan):

- **`flow/balance.ts` has ZERO production diff since Story 4.1** (a single commit in its history) ŌĆö the
  true frozen compute + sign authority. The kWŌåÆvisual math in `renderer.ts` (`edgeVisual`) is likewise
  math-unchanged.
- **`flow/model.ts` carries only the additive multi-instance identity seam** ŌĆö `git diff` of Story 4.2 ŌåÆ
  9.7 shows the *only* logic change is adding optional `FlowInput.id`, then `const id = input.id ?? input.role`,
  keying `FlowNode.id`/`FlowEdge.from` off `id`. The balance math `BUS_ORIENTATION[input.role] * input.kW`
  is untouched, so a single instance (`input.id === undefined`) yields **byte-identical** pre-9.7 output.
- **A new energy node/role is a registry + component-metadata edit ŌĆö NEVER a `balance.ts`/`buildFlowModel`-math
  edit.** Adding to `ROLES` makes the compiler enumerate exactly the role-keyed tables
  (`FUNCTION_KEYS`/`BUS_ORIENTATION`/`POWER_KEY`/`NODE_COLOR`/`NODE_ICON`) to fill ŌĆö the
  typecheck cascade is the witness that balance/model math is never named. `generator` (Story 9.14) is the

## 7. Vehicle Controls & Panels (Epic 5)

The control surface declares **two opposite feedback contracts, once each**:

- **Optimistic-then-reconcile** (`quick-actions.ts` is canonical; generalized to boolean/number/enum in
  `panel-climate.ts`, reused in `panel-media.ts` and `powerwall.ts`): the UI flips optimistically, then
  **reconciles by re-deriving from the live predicate** every `hass` tick (never a tap-time snapshot). The
  sighted render follows the optimistic value but the **SR name** (`aria-pressed`/`aria-label` via
  `srState`) follows the settled real state. The honesty fence is the exported **`RECONCILE_TIMEOUT_MS`
  (10 s)** ŌĆö a bounded `setTimeout`, cleared on reconcile and `disconnectedCallback`. *Import it; never
  redefine the magic number* (`quick-actions` exports the one constant; `panel-climate`,
  `panel-media`, and `powerwall` import it).
- **Fire-and-forget** (`commands.ts`): a one-shot `button.press`, no optimistic flip / reconcile /
  timer. Read-only panels (tires/location), physical-closure actions, and media prev/next are also
  deliberately non-optimistic (an optimistic flip would be false certainty).

**The three glanceable charge states** are the *hero's* derived read, not a raw dialect value:
`hero._chargeVisual()` maps `normalizeChargingState(charging_status)` ŌåÆ **`charging` / `plugged` / `parked`**
(`unknown`/absent ŌåÆ **Parked** unless the physical `charge_cable` sensor corroborates **plugged** ŌĆö never a
fabricated state). Note `normalizeChargingState` itself returns the finer `charging|starting|stopped|complete|disconnected|no_power|unknown`;
the parked/plugged/charging triad is the hero's collapse of that. Live kW is a direct NaN-safe `charger_power`
read ŌĆö the Hero holds **no copy** of the flow sign convention.

**Stateless `button` domain.** `button` entities have no meaningful state (`'unknown'` until first
press, reset every HA restart). Commands degrade on **missing only** (`isMissing` = `undefined`/`'unavailable'`),
**never** `isUnavailable`/`UNAVAILABLE_STATES` (which include `'unknown'` and would disable every never-pressed
command, including wake, on a fresh install).

**Wake citizenship is a CI-blocking invariant** (`data/wake.ts`): `observedWakeState`/`canWake` never
issue `button.press('wake')` under an observed `online`/`waking` state. Classification order is load-bearing
(`online ŌåÆ waking ŌåÆ asleep ŌåÆ unknown`); `waking` is checked **before** `asleep` so the post-press tick
rate-limits frantic taps. The `online` signal requires a *fresh* `'on'` (a stale `'on'` degrades to
`unknown`), so an explicit user wake of a genuinely asleep car is never blocked. The gate holds no clock
(recompute-on-render); per-instance `config.wake_cooldown` (default 60 s) rate-limits implicit wakes.

**Honest freshness is multi-dimensional** ŌĆö value, tone, AND computed baseline can each independently
lie ("never a false closed" is the one unforgivable copy error). `panel-closures` refuses "All closed"
unless every closure is `available && closed`; even a permitted stale "All closed" drops its confident
green to a `dim` tone. Stale/asleep surfaces show last-known + an "updated N ago" stamp via
`keyAgeHint`/`formatAgeHint` and `referenceNow(hass)` (the max server stamp, computed once per render) ŌĆö
**never** `Date.now()`. (Media is the inverse: no stamp ŌĆö a stale player degrades to a calm empty state.)
Peer-derived thresholds (tires) use the **fresh-corner subset only** so a stale reading can't inflate the
baseline. Cover/lock reads go through `normalizeCoverState`/`normalizeLockState` (`data/dialect`), never
inline `=== 'open'`/`'locked'`.

Shared primitives are built where first consumed, extracted where first shared: `tc-slider`
(`slider.ts`, commits `value-changed` on pointer/key **release only** ŌĆö never per drag tick, to respect
the metered Fleet API) and `statTile` (returns `nothing` when its entity is missing ŌĆö fewer tiles, not a
wall of dashes) land in charging and are reused across panels.

## 12. Enhanced Configuration ŌĆö the no-YAML editor & customization (Epic 9 / CAP-5)

Epic 9 is the **1.0.0 net-new capability**: a complete no-YAML setup / theming / customization

### 12.1 The GUI editor (`src/editor.ts`, ~2,656 LOC, `@customElement('tesla-card-editor')`)

- **5-step skippable wizard** `detect ŌåÆ confirm ŌåÆ appearance ŌåÆ tune ŌåÆ finish` (`WIZARD_STEPS`), plus a
  **normal form** (discovery summary ŌåÆ appearance ŌåÆ tune ŌåÆ name/image ŌåÆ Scene-node customization ŌåÆ "Run
  guided setup" re-entry). One shared component renders each surface in **both homes**; `_wizardActive`
  picks the home (an explicit override, else `setup_complete`, else "is this a bare stub?").
- **Persistence is per-step partial-config-to-Lovelace, NEVER `localStorage`/component state.** Resume
  marker = additive `setup_complete?` (`true` = done/normal-form; `false` = in-progress/resume; absent =
  auto-open only for a bare stub). Resume is derived in `willUpdate` (guarded by a one-shot `_resumed`
  flag so HA echoing your own `config-changed` doesn't reset within-session nav), never `render()`. The
  **resume promise (D8):** survives editor reopen, browser refresh, and cross-device ŌĆö each step persists a
  forward-compatible (R9) config; the auto-detected config is written at/after **Step 1 (Detect)** so a
  hard refresh before first save can't discard a brand-new card.
- **`_emit` vs `_patch` discipline:** `_emit(config)` = whole-config **REPLACE** + `config-changed` ŌĆö
  use whenever a key may be **deleted** (a merge can't delete). `_patch(patch)` = shallow **merge** ŌĆö
  only for pure SETs (in practice, narrowly: the `setup_complete:false` Detect write). **Every reset/clear/default
  DELETES the key** (a removed key, never a blanked value) and prunes emptied containers by
  `Object.keys(x).length > 0` (preserving unknown/future keys). A `safeClone` guard normalizes a malformed
  hand-edited container to `{}` so it never throws.

### 12.2 Editor `hass.states` boundary (Story 9.10 / decision D7)

The Epic-7 "editor reads no `hass.states`" rule is deliberately **relaxed for the editor only**. The
editor MAY read its **own** `hass` for discovery: registry maps (`hass.config.components ŌåÆ hass.devices
ŌåÆ hass.entities`) for **presence**, and `hass.states[id]` **only to annotate liveness**. Guardrails:
public `hass` surface only; **NEVER `import` an HA-frontend `src/data/*` helper** (that breaks the
bundle); keyŌåÆentity resolution still flows through the card's unchanged `data/` resolvers. Discovery is
re-derived fresh per editor open (never persisted) ŌĆö only resolved entity-IDs persist. The exception is
a **self-invalidating** `'src/editor.ts'` baseline entry in `no-bare-hass-states.mjs` (the gate FAILS if
`editor.ts` ever stops containing a bare read ŌĆö the boundary ratchets shut, never reopens). The four-state
liveness vocabulary, computed by `_liveness` off `hass.entities` (the registry) + `hass.states`:
**Ō£ō online ┬Ę ŌÜĀ unavailable ┬Ę ŌÜĀ no data yet (`unknown`) ┬Ę ŌĆö not found** (Ō£ō attests *reachable*, not *awake*;
the "not found" word sits at `--tc-text-dim` for AA, never `--tc-text-mute`).

### 12.4 "My Home" node customization (9.1ŌĆō9.4, 9.7, 9.8, 9.15) ŌĆö `energy.nodes`

- **Hide at the MODEL seam, reorder/promote at the RENDER seam.** Hide forces `kW:undefined` at
  `flowInputsFrom` (a shared `absentInput(role)` makes "hidden == absent" structural) so `buildFlowModel`
  emits `present:false` ŌĆö filtering in `render()` would desync the SVG bus from the grid (forbidden).
  Reorder is the grid-row packing (`_orderedRows`); the bus follows because `gatewaySegments` sorts taps
  by spatial anchor centre-x. Precedence: **hide wins over order.** The keyspace is `Role` (incl. vehicle),
  never `EnergyRole`.
- **Cross-row promotion (9.15)** ŌĆö `energy.nodes.rows` relaxes the within-row default. **Bus sign ŌēĀ
  row:** a node's bus contribution is its own measured net ├Ś `BUS_ORIENTATION[role]`, row-invariant by
  construction (balance never reads the rendered row). A single `rowOf(role, cfg)` classifier drives both
  render and geometry so they can't disagree.
- **Multi-instance / per-instance identity (9.7) ŌĆö `flow/instances.ts`.** Node identity is computed
  `instanceId = count <= 1 ? role : ${role}:${i+1}` ŌĆö **bare for single = zero-diff**, `:1`/`:2` only
  for genuine duplicates. The id is threaded through every seam that assumed `id===role` (`FlowNode.id`,
  the anchor map, the `edgeById` edge lookup, `scene-bus` chip lookup, focus coupling). The `instances` schema is a
  descriptor list `Partial<Record<Role, InstanceSpec[]>>` (`{ title?, entities?, config? }`); array
  length = the instance count; `config?` is consumed only for the `vehicle` role (a 2nd car's embedded
  card). N instances are consumed at the binding seam (`flowInputsFrom` flat-maps) ŌĆö `balance.ts` is
  **not** edited (it already aggregates by node id). **WRAP overflow (design decision D15 ŌĆö overflow
  wraps; never scrolls or clamps):** a row band > `WRAP_MAX_PER_ROW
  = 3` drops extras to a 2nd sub-row offset **`230px`** (`= (380 + 80)/2`, **not** the mock's 182px) into
  the bus channels (`SAFE_BAND_MAX = 6`); the overflow sub-row uses CSS `order:-1` visually while DOM/Tab
  order stays primaryŌåÆoverflow LŌåÆR; disambiguate duplicated roles by card **title**, never a numeric `:n` badge.
- **Detected-but-hidden advisory** rides the rendered card (`my-home.ts`), not the editor ŌĆö an
  "opposite-of-hide" probe with `role="status"`/`aria-live="polite"` (never red, never auto-un-hides);
  global toggle `notify_hidden_detected?` (default-on).

## 13. Key Constraints & Gotchas

- **`useDefineForClassFields: false`** is load-bearing (see ┬¦2). Never "fix" it.
- **No hard-coded Tesla entity IDs** ŌĆö resolve by function-name (see ┬¦5). Energy ids auto-detect too.
- **One sign convention per layer, never copied** ŌĆö canonical (`flow/balance.ts`), RAW in ecosystem
  cards/panels, and Scene ribbon/bus VIEWs of `computeBalance().net`; the rawŌåÆcanonical flip happens once,
  at `data/dialect` (┬¦6, ┬¦8).
- **`data/ ŌåÉ flow/ ŌåÉ components/`** direction is gate-enforced; only `src/data/` reads `hass.states` at
  runtime (only `data/history.ts` reaches recorder history, via `callWS`); the **editor** is the D7
  exception (┬¦3, ┬¦12).
- **Two token gates** ŌĆö every `var(--tc-*)` carries a fallback **and** names a real token (┬¦9).
- **The frozen authority is `flow/balance.ts` (AR-6), not "all six flow files"** ŌĆö `flow/` is now 7 files;
  a new node/role is registry + component metadata; `model.ts` carries only the `id ?? role` seam (┬¦6.1).
- **Reuse the Epic-5 feedback machines** ŌĆö import `RECONCILE_TIMEOUT_MS`, never redefine it.
- **Keep the Lovelace contract** (`setConfig`/`getCardSize`/`getStubConfig`/`getConfigElement`); honour
  the forward-compat spread.
- **Every Epic-9+ config key is additive and zero-diff-when-absent**; in the editor, reset/clear/default
  **deletes** the key via whole-config `_emit` (a merge can't delete) ŌĆö never a blanked value (┬¦12.1).
- **`variant: 'compact'` is presentation-only** ŌĆö it never force-hides sections; compact layout is the
  element-scope `:host([compact])` host attribute, never a viewport `@media` (┬¦8.1).
- **Never commit `dist/`** ŌĆö gitignored, built in CI. **Version sync:** `package.json` `version` Ōåö
  `src/const.ts` `CARD_VERSION` (`0.2.0`) Ōåö git tag must all match (the `version-sync` gate).
- **Standalone public repo** ŌĆö commit card changes *inside* `tesla-card/`. BMAD planning artifacts live
  in the sibling private `tesla-card-planning/` repo (gate-blocked here).
- **No vehicle artwork ships** ŌĆö the bundled default is a generic EV silhouette; bring your own layer
  packs (HA-served from `/local/tesla-card/<model>/`, never committed). See
  [`asset-packs.md`](./asset-packs.md) / [`trade-dress.md`](./trade-dress.md).
- **Sanctioned hard-coded colours** (the only departures from `--tc-*` tokens): the `panel-location`
  `135deg` map gradient (`#1b2533`/`#0f1620`), the `tc-slider` `#fff` thumb, the Scene `GATEWAY_STROKE`
  `#cfe2ff`, the generic-EV SVG art, and the aperture skins. All others are tokens.
