## 1. Executive Summary

`tesla-card` is a Tesla-app-style custom [Lovelace](https://www.home-assistant.io/dashboards/)
card for Home Assistant, built in **TypeScript (strict) + [Lit](https://lit.dev) 3**. It renders a
single vehicle as a centered **living hero** (recolorable inline-SVG car with charge/aperture state)
over a **tab bar** of feature panels (charging, climate, energy, closures, tyres, location, media),
plus quick-action toggles and command buttons. Beyond the vehicle card it also ships a set of
standalone **ecosystem cards** (solar / powerwall / grid / home / wall-connector / **generator**) and a
composed **"My Home" energy Scene**, all from the same bundle.

Two capability arcs define the current shape:

- **Live, honest energy flow.** A single sign/unit convention and a balance authority drive both the
  Hero charge overlay and the Scene's Gateway bus, so the visuals never disagree with the numbers and
  never animate a false flow (Epics 4/6/8).
- **No-YAML Enhanced Configuration (Epic 9 / the 1.0.0 net-new).** A ~2,400-line HA-native GUI editor
  delivers a guided first-run wizard + a normal form for theming, per-entity overrides, and "My Home"
  Scene customization (hide / reorder / cross-row promote / duplicate nodes), plus a brand-new
  `generator` node **type** — all additive and zero-diff-when-absent (§13).

It ships as a **single bundled ES module** (`dist/tesla-card.js`) consumed by Home Assistant as a
Lovelace resource and distributed via [HACS](https://hacs.xyz/). There is **no application server,
no API layer, and no datastore** — all state comes from the HA `hass` object injected by the
dashboard at runtime, every service call goes back through `hass.callService`, and the one historical
read rides `hass.callWS` (HA's own authenticated WebSocket — never a socket of the card's own).

The card's robustness rests on **entity resolution by stable function-name.** Tesla Fleet integrations
expose ~84 entities whose IDs are *not* uniformly prefixed (~30 live on the bare device, e.g.
`sensor.odometer`, `cover.sunroof`). The card resolves entities by stable *function-name* keys so it
works across installs, device renames, and integration prefixes without hard-coded IDs.

## 3. Layered Architecture — `data/ ← flow/ ← components/`

The source is organized into three layers with a **one-way dependency direction**, enforced by the
`no-cycle` and `no-bare-hass-states` lint gates:

```
            ┌─────────────┐    reads hass.states (the ONLY layer allowed to)
   data/    │  freshness  │    dialect · energy · history · registry · resolve · slice · wake
            └──────┬──────┘    pure read-model + normalization boundary
                   │  (canonical, sign-normalized data)
            ┌──────▼──────┐
   flow/    │   balance   │    model · binding · renderer · hero-svg · scene-bus · my-home · instances
            └──────┬──────┘    pure energy-flow math; imports data/, never components/
                   │  (FlowModel + visual derivations)
            ┌──────▼──────┐
components/ │  tc-* + helpers │  hero/panels/quick-actions/commands/ecosystem/scene + car/weather/chart/node-hero
            └─────────────┘    Lit elements + render helpers; import flow/ and data/
```

- **`src/data/`** is the data-access boundary. `data/freshness.ts` is the **sole reader of
  `hass.states`** (`read`/`readKey`/`readRaw`, `referenceNow`, staleness classification); every other
  runtime module reads through it. `data/dialect.ts` quarantines per-integration spelling differences
  (`normalizeChargingState`/`normalizeCoverState`/`normalizeLockState`, the `DIALECTS` table).
  `data/history.ts` is the **one** module that reaches HA recorder history — over
  `hass.callWS('history/history_during_period')`, **not** `hass.states` — so the boundary still holds.
- **`src/flow/`** is pure energy-flow logic. It imports `data/` (and root leaves) and **never**
  `components/`. `flow/balance.ts` owns the single sign/unit convention; `flow/instances.ts` (Epic 9)
  is the DOM-free per-instance identity helper.
- **`src/components/`** holds the Lit elements (`tc-*`) and pure render helpers (`car.ts`,
  `weather-vignette.ts`, `chart.ts`, `node-hero.ts`). It imports `flow/` and `data/`.
- **Root leaf modules** (`strings.ts`, `log.ts`, `layer-contract.ts`) import nothing upward, keeping
  the cycle check trivially green. All user-facing copy lives in `strings.ts`; all `console.*` routes
  through `log.ts`.

> **The editor is the one sanctioned boundary exception (D7).** `src/editor.ts` MAY read its **own**
> `hass` for discovery (registry maps for presence, `hass.states[id]` only to annotate liveness) — a
> self-invalidating baseline entry in `no-bare-hass-states` records it. The card **runtime** never does
> (see §13). It still must never `import` an HA-frontend `src/data/*` helper.

## 4. Component Tree (the `components/` layer)

A single top-level custom element orchestrates a flat set of children. There is **one**
`@customElement('tesla-card')`; every child is a `tc-*` element and (with a few exceptions) extends a
shared base class. Components communicate **downward** by props and **upward** by `CustomEvent`.

```
tesla-card  (src/tesla-card.ts)  — LitElement, implements LovelaceCard
│  state: hass, _config, _resolvedConfig (memoised), _energy (resolved energy), _panel (active tab)
│  owns: design tokens on :host, tab bar, @open-panel routing, customCards registration
│
├── tc-hero            components/hero.ts          living car (paint + charge + apertures) + battery + flow overlay
├── tc-quick-actions   components/quick-actions.ts optimistic toggles (lock/climate/port/sentry/…)
├── tc-commands        components/commands.ts      fire-and-forget command buttons (wake/honk/flash/…)
└── tc-panel-*  (the active tab; exactly one in the DOM at a time)
    ├── tc-panel-charging   panel-charging.ts   ── uses ─▶ tc-slider
    ├── tc-panel-climate    panel-climate.ts
    ├── tc-panel-energy     panel-energy.ts     (Energy tab — inserted only when a site is detected)
    ├── tc-panel-closures   panel-closures.ts
    ├── tc-panel-tyres      panel-tyres.ts
    ├── tc-panel-location   panel-location.ts
    └── tc-panel-media      panel-media.ts      ── uses ─▶ tc-slider

Standalone ecosystem cards (own dashboard cards; share EcosystemCard shell; use chart/node-hero):
    tc-solar · tc-powerwall · tc-grid · tc-home · tc-wall-connector · tc-generator
"My Home" Scene element:
    tc-my-home  (components/my-home.ts) — live SceneBus render of the energy Scene (+ embedded vehicle cells)

tesla-card-editor  (src/editor.ts)  — LitElement, implements LovelaceCardEditor (lazy-loaded, ~2,400 LOC)
```

**Element count.** Loading the bundle registers **20** custom elements total: `tesla-card`,
`tesla-card-editor`, and **18 `tc-*`** elements. Of those, **8 are Lovelace picker cards** registered
in `window.customCards` — `tesla-card` + `tc-solar` / `tc-powerwall` / `tc-grid` / `tc-home` /
`tc-wall-connector` / `tc-generator` / `tc-my-home`. The editor is **not** registered at load (lazy).
`car.ts`, `weather-vignette.ts`, `chart.ts`, `node-hero.ts`, and `ecosystem-card.ts` register **no**
element (render helpers / base class).

**Base class — `TcBase` (`src/base.ts`).** Extends `LitElement` and supplies exactly two reactive
properties to every vehicle child:

```ts
@property({ attribute: false }) public hass?: HomeAssistant;
@property({ attribute: false }) public config!: TeslaCardConfig;   // the RESOLVED config
```

Children read `config` (not `_config`). The non-`TcBase` elements are **`tc-slider`** and
**`tesla-card-editor`** (extend `LitElement` directly), and the **ecosystem cards** + **`tc-my-home`**,
which extend `EcosystemCard` / `LitElement` because they are standalone surfaces, not vehicle children.

**Registration.** Children are pulled in by **side-effect imports** in `src/tesla-card.ts`
(`import './components/hero';` …) — each imported module runs its own `@customElement(...)` on load.
`tc-slider` is side-effect-imported by its consumers (`panel-charging.ts`, `panel-media.ts`,
`powerwall.ts`), not by the parent. To add a child component, define its `@customElement` and add a
side-effect import where it is used. Assert registration against the **bundle entry**
(`tesla-card.contract.test.ts`), not a convenient internal module.

**One panel in the DOM.** The parent renders **one panel at a time** — inactive panels are *absent
from the DOM*, not hidden, so there is no cross-panel focus trap and no tabbable hidden content. The
**Energy tab** `splice`-inserts at **index 2** (after Charging) only when `hasEnergySite(_energy)`,
and without reflowing the user out of their current panel. `default_panel` sets the first tab and
falls back to the first available panel if it names an absent/hidden one.

## 5. Entity Resolution (the `data/` layer)

Resolution lives under **`src/data/`** (the catalog seed is in `src/const.ts`). It runs **once** in
the parent and the resolved map is passed down; children never re-resolve.

### 5.1 The catalog — `DEFAULT_ENTITIES` (`const.ts`)
An `as const` object mapping **84 stable function-name keys** (e.g. `battery_level`, `odometer`,
`time_to_full_charge`, `tire_fl`, `seat_rl`, `wake`, `boombox`) to the exact live entity IDs of a
reference "Garage Model Y" install. `EntityKey = keyof typeof DEFAULT_ENTITIES`. There is **no
`DEFAULT_IMAGE`** — the old `/local/model_y.png` fallback was removed so a fresh install never 404s;
the zero-config default hero is the bundled inline-SVG generic EV.

> ⚠️ **Tesla Fleet IDs are NOT uniformly prefixed.** ~30 of the keys point at *bare* device
> entities — `sensor.odometer`, `cover.sunroof`, `sensor.shift_state`, all tyre pressure sensors,
> rear seat heaters, `sensor.speed`, `sensor.power`, climate cabin-overheat, … The rest carry the
> `garage_model_y_` device prefix. Blindly prefixing the bare ones yields `unavailable`.
> `DEFAULT_ENTITIES` encodes the exact mix.

### 5.2 Resolution flow (`data/resolve.ts`, `data/registry.ts`)
The function-key vocabulary (`ROLES`, `FUNCTION_KEYS`, `BUS_ORIENTATION`) is the single source of
truth in `data/registry.ts` — **84 vehicle + 22 energy keys** (the value tables in `const.ts` and
`energy.ts` are typecheck-bound back to these names, so the three cannot diverge). `resolveEntities()`
builds a complete id map for every key, in order of precedence: explicit `config.entities[key]`
override → registry match by canonical identity within the detected Tesla device (any prefix,
`TESLA_PLATFORMS` set) → direct prefix guess against live `hass.states` → bare global id if present →
bundled `DEFAULT_ENTITIES[key]` fallback. Without `hass`, only overrides + defaults are used. Degrades
gracefully when the entity/device registry is absent (older HA, demo harness).

### 5.3 Energy resolution (`data/energy.ts`)
Energy/Wall-Connector entities are a **second, separate** resolution. They match by **stable
function-slug substring in the object-id** (e.g. any `*solar_power*` sensor) — prefix-independent and
tolerant of the integration's `_2` duplicate-entity suffix (shortest matching object-id wins).
`resolveEnergyEntities → _energy` runs once in the parent alongside the vehicle config;
`hasEnergySite(_energy)` gates the Energy tab. `config.energy.entities` overrides always win;
`config.energy.hide: true` suppresses the panel even when a site is detected. Read via NaN-safe
`numById`/`stateById`/`unitById` (units read live — the Wall-Connector handle temperature is **°F**
on the reference install)/`attrById` (array/non-string attrs, e.g. `select` options).

### 5.4 Memoisation & propagation
`TeslaCard` memoises resolution on the identity of `hass.entities` / `hass.devices` / `_config`,
builds `_resolvedConfig = { ..._config, entities: resolveEntities(...) }`, and passes that resolved
`cfg` (+ `hass`) to every child. Children read ids via `helpers.ts` (`entityId`, `rawState`, …),
which read `config.entities[key]`.

> 🔑 **Rule:** never hard-code a Tesla entity ID (vehicle **or** energy) in a component. Resolve by
> stable function-name through the parent's resolved config so the card works across installs/prefixes.

## 6. Live Energy Flow (the `flow/` layer)

The energy-flow engine is the card's most-verified subsystem. **One sign/unit convention, declared
exactly once** in `flow/balance.ts`: **kW everywhere; battery + = charging; grid + = import**; an
edge's `from→to` is its positive-flow sense. Every surface that infers charge direction consumes this
one authority — there is **no private copy**.

- **Normalization at the boundary.** Raw entity signs (some batteries report `battery − = charging`)
  are flipped to canonical at the **`data/dialect`** boundary *before* the FlowModel sees them, so the
  model only ever sees `battery + = charging`.
- **`computeBalance` (`flow/balance.ts`)** is the sole balance authority — per-node net + per-edge
  balance, role-generic (no per-node-type branches). It is the #1 verification target: property tests
  assert sign-convention correctness **and** conservation (per-node sum-to-zero). `IDLE_KW`/deadband
  `= 0.05` kW: a sub-deadband or NaN/`unavailable` read makes an edge **quiescent** (present,
  `direction:'none'`), never broken — sensor jitter never animates as flow.
- **`bindFlowModel(hass, config)` (`flow/binding.ts`)** auto-detects the **six** energy roles
  (`ENERGY_ROLES`) via the `data/energy` resolver; `POWER_KEY` maps each role to its single `*_power`
  sensor; `provenance ∈ {measured|inferred|quiescent}` tags each edge. The `flowInputsFrom` seam is
  where node **hide** (force `kW:undefined`) and **multi-instance** (flat-map role→N inputs) are
  applied — never a render-only filter.
- **`FlowRenderer { update(model) }` (`flow/renderer.ts`)** is the renderer seam. Two implementations
  share **one** kW→visual helper (`edgeVisual`/`edgeVisuals`, `NODE_COLOR`/`NODE_ICON`):
  - `HeroSvgRenderer` (`flow/hero-svg.ts`) — node-id → the fixed **1024×687** Hero viewBox.
  - `SceneBusRenderer` (`flow/scene-bus.ts`) — node-id → live `getBoundingClientRect()` anchors.
  The two differ **only** in coordinate source; a test deep-equals their visuals.
- **Colour-blind safe:** every edge carries a node-anchored label + kW magnitude (source is never
  hue-only). **Reduced-motion → a legible static read** (arrowheads + kW labels, no animation).

The recognizable flow states (`src/fixtures/flow-*.json` — grid import / export, solar surplus,
charging vs plugged-idle, storm-watch/islanding, vampire drain) are each a **committed fixture + a
FlowModel assertion** (Epic 9 adds `energy-generator.json`).

> 🔗 **Composed-view authority split:** the discrete `charging_status` entity owns the Hero halo
> (`hero._chargeVisual()`); the FlowModel owns the wall-connector edge magnitude/direction — the WC
> edge **is** the car-charging edge (the vehicle is **not** a flow node). A test asserts halo and edge agree.

### 6.1 The frozen authority — AR-6 (replaces the old "FR-33 = six frozen files")

The earlier rule that *all six `flow/` files* were frozen is **retired.** Epic 9's new role and
multi-instance support deliberately edited `binding.ts` / `renderer.ts` / `hero-svg.ts` / `scene-bus.ts`
and added one additive seam to `model.ts`. The surviving, precise invariant is **AR-6**:

- **`flow/balance.ts` has ZERO production diff since Story 4.1** — the true frozen compute + sign
  authority. The kW→visual math in `renderer.ts` (`edgeVisual`) is likewise math-unchanged.
- **`flow/model.ts` carries only the additive multi-instance identity seam** (`const id = input.id ?? input.role`,
  Story 9.7 — single-instance output is byte-identical).
- **A new energy node/role is a registry + component-metadata edit — NEVER a `balance.ts`/`buildFlowModel`-math
  edit.** Adding to `ROLES` makes the compiler enumerate exactly the role-keyed tables
  (`FUNCTION_KEYS`/`BUS_ORIENTATION`/`POWER_KEY`/`NODE_COLOR`/`NODE_ICON`/`NODE_XY`) to fill — the
  typecheck cascade is the witness that balance/model math is never named. This is how `generator`
  (Story 9.14) was added (§13).

## 7. Vehicle Controls & Panels (Epic 5)

The control surface declares **two opposite feedback contracts, once each**:

- **Optimistic-then-reconcile** (`quick-actions.ts` is canonical; generalized in `panel-climate.ts`,
  reused in `panel-media.ts` and `powerwall.ts`): the UI flips optimistically, then **reconciles by
  re-deriving from the live predicate**. The sighted render follows the optimistic value but the **SR
  name** follows the settled real state. The honesty fence is the exported **`RECONCILE_TIMEOUT_MS`
  (10 s)** — a bounded `setTimeout`, not polling. *Import it; never redefine the magic number.*
- **Fire-and-forget** (`commands.ts`): a one-shot `button.press`, no optimistic flip / reconcile /
  timer. Read-only panels (tyres/location) and physical-closure actions are also deliberately
  non-optimistic (an optimistic flip would be false certainty).

**Stateless `button` domain.** `button` entities have no meaningful state (`'unknown'` until first
press, reset every HA restart). Commands degrade on **missing only** (`isMissing`), **never**
`isUnavailable` — which would disable every never-pressed command (including wake) on a fresh install.

**Wake citizenship is a CI-blocking invariant** (`data/wake.ts`): `observedWakeState`/`canWake` never
issue `button.press('wake')` under an observed `online`/`waking` state. The `online` signal requires a
*fresh* `'on'` (a stale `'on'` degrades to `unknown`), so an explicit user wake of a genuinely asleep
car is never blocked. Per-instance `config.wake_cooldown` (default 60 s) rate-limits implicit wakes.

**Honest freshness is multi-dimensional** — value, tone, AND computed baseline can each independently
lie ("never a false closed" is the one unforgivable copy error). Stale/asleep surfaces show last-known
+ an "updated N ago" stamp via `keyAgeHint`/`formatAgeHint` and `referenceNow(hass)` — **never**
`Date.now()`. Cover/lock reads go through `normalizeCoverState`/`normalizeLockState` (`data/dialect`),
never inline `=== 'open'`/`'locked'`.

Shared primitives are built where first consumed, extracted where first shared: `tc-slider`
(`slider.ts`, commits `value-changed` on pointer/key **release only** — never per drag tick) and
`statTile` (hides when its entity is missing) land in charging and are reused across panels.

## 8. Ecosystem Suite & "My Home" Scene (Epics 6 & 8)

- **Shared shell, extended not forked.** `ecosystem-card.ts` is an abstract (unregistered)
  `EcosystemCard extends TcBase` exposing `renderShell` + `ecosystemShellStyles` + the pure
  `accentVar` — the one place accent hex is composed. `tc-solar` / `tc-powerwall` / `tc-grid` /
  `tc-home` / `tc-wall-connector` / `tc-generator` each "register + component only." Grid uses the
  `'neutral'` shell accent (not one of the 8 `ACCENT_SEMANTICS` keys). Cards are **presence-tolerant**
  (render standalone or composed) and coordinate only via the **shared `hass`** — never inter-card
  messaging.
- **Ecosystem cards read RAW sensor signs**, lifted verbatim from `panel-energy.ts` (grid `+=import`,
  battery `−=charging`, `wc_power`/`generator_power` non-negative magnitudes, 0.05 kW deadband) — **the
  opposite** of the FlowModel's canonical sign. *Never copy a sign convention*; each sign is pinned so
  a flip fails CI.
- **The weather vignette reads HA CORE, never Tesla** — `weather.home` + `sun.sun` through the
  sanctioned `readRaw` boundary (same as `PaintSource`). It is a `weatherVignette()` render-helper
  (no element); reduced-motion freezes it to a static read; it carries no age stamp.

**Epic 8 depth** added detail shells + stat grids + deep-links, **per-node hero art** (`node-hero.ts`,
4 kinds — Solar reuses the weather vignette), **inline history charts** (`chart.ts` over
`data/history.ts`), the **Powerwall control surface** (`operation_mode` select + `backup_reserve`
slider, reusing the Epic-5 reconcile contract verbatim — `RECONCILE_TIMEOUT_MS` *imported*), and the
**Vehicle presentation cell** in the Scene. The Vehicle is the Scene's embedded **card** but **NOT a
flow node**: a pure presentation cell fed by `wcVehicleEdge(model)` (`Math.abs(WC edge kW)`).

**The Gateway bus + self-powered ribbon** are pure **VIEWS** of the one `computeBalance(model).net`
(canonical sign), so **bus and ribbon agree by construction.** `gatewaySegments` walks taps left→right
by anchor centre-x; `ribbonTiles`/`selfPowered` aggregate the same net; `pct` degrades to `—` under
sub-`IDLE_KW` load (never a false `0%`/`100%`). Focus coupling is **computed** (`coupledRoles` by net
sign), not hard-coded.

> 🧊 **The composed-Scene views live in the non-frozen `flow/my-home.ts`** (`wcVehicleEdge`,
> `gatewaySegments`, `selfPowered`/`ribbonTiles`, `coupledRoles`, `RafCoalescer`, the trunk/axis
> constants `BUS_WIDTH_MAX`/`SCENE_PHONE_MAX`, and the Epic-9 WRAP geometry). They are VIEWS of
> `computeBalance().net` — **not** a second sign convention, **not** a second balance authority.

- **`SceneBusRenderer` is wired LIVE** through the Story-4.4 `setAnchors` seam (`components/my-home.ts`
  feeds real `getBoundingClientRect()` rects — "only the coordinate source differs"). Reflow
  lifecycle: `ResizeObserver`/`IntersectionObserver` → a debounced `RafCoalescer` recompute → cached
  geometry → pure-CSS dash motion → all torn down in `disconnectedCallback`. The slice-gate
  (`data/slice.ts`) watches the **union of every entity the children read** (not just `*_power`).
- **Arbitrary topology is free-by-construction:** present-gating + centred row-groups pack present
  cards with zero ghost cells (`axisForWidth`/`SCENE_PHONE_MAX` move only the axis selection). The
  **half-alive Scene** (live local energy + asleep car) is the NORMAL state — calm, not broken.

## 9. Design System (tokens & styles)

Defined once in **`src/styles.ts`**: `tokens` (a `:host { … }` block of `--tc-*` custom properties set
on the `tesla-card` host) and `sharedStyles` (reusable class rules + the locked a11y keyframe corpus
`{tc-pulse, tc-shimmer}`). CSS custom properties inherit across shadow DOM, so **children import only
`sharedStyles` + their own `css`** and read tokens via `var(--tc-…)`.

> 🔒 **Two hard token gates (`styles.test.ts` + `token-defined.mjs`):**
> 1. Every `var(--tc-*)` consumption MUST carry the DESIGN.md value as a **fallback** (`var(--tc-blue, #38bdf8)`).
> 2. **AND** the token must be a **real token** declared in `styles.ts` (or set locally) — the
>    `token-defined` gate (Epic 9) closes the Epic-8 blind spot where a fallback *existed* but the token
>    did not (two shipped bugs). The fallback gate proves a fallback exists; the token-defined gate
>    proves the token is real. **Both must pass.** Also: the "exactly one 180deg" gate is comment-blind —
>    write `180°` in prose, not `180deg`.

`styles.ts` also exports machine-checkable contract maps: **`ACCENT_SEMANTICS` (8 accents → meaning:
blue/green/amber/red/purple/orange/teal + `copper` `#c2855b` = generator/fuel)**,
`INTERACTION_PRIMITIVES`/`INTERACTION_BANS`, `BREAKPOINTS` (a **build-time TS constant**, tree-shaken —
CSS `@media` can't read custom properties), `FRESHNESS_STATES`, and **`LIGHT_TOKENS`** (Epic 9 — the
card-only light palette, single-sourced for both the `:host([theme='light'])` override block and the
editor's live preview). The pattern: an exported `as const` map is the machine-checkable half; comments
are the human half.

See [`component-inventory.md`](./component-inventory.md) for the full token, primitive, and helper
tables, and [`recolorable-body.md`](./recolorable-body.md) / [`layer-contract.md`](./layer-contract.md)
for the recolor pipeline and the published `@unstable` layer contract.

## 10. Lovelace Card Contract

Implemented on `TeslaCard` in `src/tesla-card.ts`:

| Method | Kind | Behaviour |
|---|---|---|
| `setConfig(config)` | instance | throws **only** on falsy config; spreads `{ ...config }` (unknown/future keys preserved); applies `default_panel` |
| `getCardSize()` | instance | returns a fixed size (16) |
| `getStubConfig()` | static | returns `{ type: 'custom:tesla-card' }` |
| `getConfigElement()` | static async | **lazy-imports** the editor (`await import('./editor')`) then creates `tesla-card-editor` |

> 🔑 **Rule:** keep this contract intact. `setConfig` is a **pinned forward-compat contract** (R9):
> both the card and the editor spread unknown keys and never throw on extras — "old YAML never breaks
> on a new build; a newer key still renders on an older build." A garbage `default_panel` falls back to
> `charging`. The lazy editor import keeps the editor out of the initial render path; Rollup
> `inlineDynamicImports: true` folds it back into the single bundle (the HACS one-file install).

## 12. Key Constraints & Gotchas

- **`useDefineForClassFields: false`** is load-bearing (see §2). Never "fix" it.
- **No hard-coded Tesla entity IDs** — resolve by function-name (see §5). Energy ids auto-detect too.
- **One sign convention per layer, never copied** — consume `flow/balance.ts` (canonical `battery + =
  charging`); ecosystem cards read RAW signs (`panel-energy.ts`); the Scene ribbon/bus are VIEWs of
  `computeBalance().net`.
- **`data/ ← flow/ ← components/`** direction is gate-enforced; only `data/freshness.ts` reads
  `hass.states` at runtime, and only `data/history.ts` reaches recorder history (via `callWS`, never
  polled). The **editor** reads `hass.states` under the D7 exception (§3, §13).
- **Two token gates** — every `var(--tc-*)` carries a fallback **and** names a real token (§9).
- **The frozen authority is `flow/balance.ts` (AR-6), not "all six flow files"** — a new node/role is
  registry + component metadata; `model.ts` carries only the `id ?? role` seam (§6.1).
- **Reuse the Epic-5 feedback machines** — import `RECONCILE_TIMEOUT_MS`, never redefine it.
- **Keep the Lovelace contract** (`setConfig`/`getCardSize`/`getStubConfig`/`getConfigElement`); honour
  the forward-compat spread.
- **Never commit `dist/`** — gitignored, built in CI. **Version sync:** `package.json` `version` ↔
  `src/const.ts` `CARD_VERSION` (`0.2.0`) ↔ git tag must all match (the `version-sync` gate).
- **Standalone public repo** — commit card changes *inside* `tesla-card/`. BMAD planning artifacts live
  in the sibling private `tesla-card-planning/` repo (gate-blocked here).
- **No vehicle artwork ships** — the bundled default is a generic EV silhouette; bring your own layer
  packs (HA-served from `/local/tesla-card/<model>/`, never committed). See
  [`asset-packs.md`](./asset-packs.md) / [`trade-dress.md`](./trade-dress.md).

## 13. Enhanced Configuration — the no-YAML editor & customization (Epic 9 / CAP-5)

Epic 9 is the **1.0.0 net-new capability**: a complete no-YAML setup / theming / customization
experience, "My Home" Scene customization, and a new energy node **type**. **Every Epic-9 config key is
additive and zero-diff-when-absent** — omitted ⇒ today's behavior byte-for-byte.

### 13.1 The GUI editor (`src/editor.ts`, ~2,400 LOC, `@customElement('tesla-card-editor')`)

- **5-step skippable wizard** `detect → confirm → appearance → tune → finish` (`WIZARD_STEPS`), plus a
  **normal form** (discovery summary → appearance → tune → name/image → Scene-node customization → "Run
  guided setup" re-entry). One shared component renders each surface in **both homes**.
- **Persistence is per-step partial-config-to-Lovelace, NEVER `localStorage`/component state.** Resume
  marker = additive `setup_complete?` (`true` = done/normal-form; `false` = in-progress/resume; absent =
  auto-open only for a bare stub). Resume is derived in `willUpdate`, never `render()`. The **resume
  promise (D8):** survives editor reopen, browser refresh, and cross-device — each step persists a
  forward-compatible (R9) config; the auto-detected config is written at/after **Step 1 (Detect)** so a
  hard refresh before first save can't discard a brand-new card.
- **`_emit` vs `_patch` discipline:** `_emit(config)` = whole-config **REPLACE** + `config-changed` —
  use whenever a key may be **deleted** (a merge can't delete). `_patch(patch)` = shallow **merge** —
  only for pure SETs. **Every reset/clear/default DELETES the key** (a removed key, never a blanked
  value) and prunes emptied containers by `Object.keys(x).length > 0` (preserving unknown/future keys).

### 13.2 Editor `hass.states` boundary (Story 9.10 / decision D7)

The Epic-7 "editor reads no `hass.states`" rule is deliberately **relaxed for the editor only**. The
editor MAY read its **own** `hass` for discovery: registry maps (`hass.config.components → hass.devices
→ hass.entities`) for **presence**, and `hass.states[id]` **only to annotate liveness**. Guardrails:
public `hass` surface only; **NEVER `import` an HA-frontend `src/data/*` helper** (that breaks the
bundle); key→entity resolution still flows through the card's unchanged `data/` resolvers. Discovery is
re-derived fresh per editor open (never persisted) — only resolved entity-IDs persist. The exception is
a **self-invalidating** `'src/editor.ts'` baseline entry in `no-bare-hass-states.mjs`. The four-state
liveness vocabulary: **✓ online · ⚠ unavailable · ⚠ no data yet (`unknown`) · — not found** (✓ attests
*reachable*, not *awake*).

### 13.3 Customization surfaces

- **Per-entity overrides (9.11)** — `ha-selector`/`ha-form` used directly in the Lit template (no JS
  import → `import-allowlist` satisfied by construction). A single `OVERRIDE_TARGET` map keys both
  discovery (read) and the picker (write): vehicle → `config.entities[key]`, energy →
  `config.energy.entities[key]`. Accordion-in-place; a `— not found` row opens an **unfiltered** picker
  ("map-a-miss"); an honest dead-pick is saved regardless with a polite ⚠ announce; reset-to-auto
  **deletes** the key.
- **Appearance & theming (9.12)** — paint swatches must carry + write the **curated HEX** from
  `PAINT_PRESETS`, not the bare key (`resolvePaint`'s literal-string branch returns a CSS keyword
  verbatim *before* presets — writing `'blue'` rendered a garish primary; the epic's one HIGH bug).
  Theme override is **card-only** (`appearance.theme?: 'light'|'dark'`, Auto = absent): a host `theme`
  attribute → a `:host([theme='light'])` token-override block re-resolves the colour tokens, single-
  sourced from `LIGHT_TOKENS` (never an HA global theme var). The hex field is a `{ text: {} }` selector
  written verbatim — never clamped/validated/substituted. Full-card **live preview** reuses the real
  `carView`, coalesced per pick.
- **Tune step + version floor (9.13 / D8)** — editor floor = runtime floor = **`2024.4.0`, no raise**
  (every chosen widget predates it; the entity-selector `filter:` form shipped HA 2023.3). The README
  caveat is **API-stability** (`ha-form`/`ha-selector` are HA-internal, non-SemVer), **not** version
  exclusion. `TyresConfig.units?: 'psi'|'bar'` is **display-only** (thresholds stay stored in the native
  unit). Tune toggles/dropdowns are per-card globals — never instance-suffixed.

### 13.4 "My Home" node customization (9.1–9.4, 9.7, 9.8, 9.15) — `energy.nodes`

- **Hide at the MODEL seam, reorder/promote at the RENDER seam.** Hide forces `kW:undefined` at
  `flowInputsFrom` (a shared `absentInput(role)` makes "hidden == absent" structural) so `buildFlowModel`
  emits `present:false` — filtering in `render()` would desync the SVG bus from the grid (forbidden).
  Reorder is the grid-row packing (`_orderedRows`); the bus follows because `gatewaySegments` sorts taps
  by spatial anchor centre-x. Precedence: **hide wins over order.**
- **Cross-row promotion (9.15)** — `energy.nodes.rows` relaxes the within-row default. **Bus sign ≠
  row:** a node's bus contribution is its own measured net × `BUS_ORIENTATION[role]`, row-invariant by
  construction. A single `rowOf(role, cfg)` classifier drives both render and geometry.
- **Multi-instance / per-instance identity (9.7) — `flow/instances.ts`.** Node identity is computed
  `instanceId = count <= 1 ? role : ${role}:${i+1}` — **bare for single = zero-diff**, `:1`/`:2` only
  for genuine duplicates. The id is threaded through every seam that assumed `id===role` (FlowNode.id,
  the anchor map, `edgeByRole`, `scene-bus` chip lookup, focus coupling). The `instances` schema is a
  descriptor list `Partial<Record<Role, InstanceSpec[]>>` (`{ title?, entities?, config? }`); array
  length = the instance count; `config?` is consumed only for the `vehicle` role (a 2nd car's embedded
  card). N instances are consumed at the binding seam (`flowInputsFrom` flat-maps) — `balance.ts` is
  **not** edited (it already aggregates by node id). **WRAP overflow (D15):** a row band > `WRAP_MAX_PER_ROW
  = 3` drops extras to a 2nd sub-row offset `230px` into the bus channels; disambiguate duplicated roles
  by card **title**, never a numeric `:n` badge.
- **Detected-but-hidden advisory** rides the rendered card (`my-home.ts`), not the editor — an
  "opposite-of-hide" probe with `role="status"`/`aria-live="polite"` (never red, never auto-un-hides);
  global toggle `notify_hidden_detected?` (default-on).

### 13.5 Generator — a new node TYPE (Story 9.14)

The witness for AR-6 (§6.1): adding `generator` to `ROLES` made the compiler enumerate exactly the
role-keyed tables to fill — `FUNCTION_KEYS.generator = ['generator_power']`, **`BUS_ORIENTATION.generator
= 1`** (source sign, like solar), `EnergyEntities.generator_power?`, `POWER_KEY`, `NODE_COLOR`,
`NODE_ICON` (`mdiGeneratorStationary`), `NODE_XY`, editor `NODE_LABELS` — and **never** `balance.ts`/
`buildFlowModel` math. Because `ENERGY_ROLES = Object.keys(POWER_KEY)` and `SCENE_NODES = ENERGY_ROLES`,
the generator auto-flows through binding→model→balance→ribbon→bus by construction. **Copper `#c2855b`**
is the 8th `ACCENT_SEMANTICS` accent; `tc-generator` is modeled on the *simple* ecosystem mold
(`grid.ts`/`home.ts`), and is the 8th `window.customCards` picker entry.
